# Ticket #395170 — UPS Label Generation Fails with Invalid Unit of Measurement for International Orders

- **Customer**: Ibrahim Hatoum / baro@thekriptstore.com (localeuropean.myshopify.com)
- **Product**: Shopify Multi Carrier Shipping Label
- **Duration**: 2026-06-15 to 2026-06-18 (4 days, ongoing)

## Timeline & Key Phases

### Phase 1: Initial Report and Triage (2026-06-15)
- Customer (Ibrahim Hatoum, Director of Operations) reported inability to generate UPS labels for orders 68951 and 68952 via the MCSL app.
- Internal agent investigated and identified error: "Invalid Unit of Measurement for the given country" — orders destined for Germany and Canada.
- Store units were set to pounds (LBS) in the app, but order details displayed in kg/cm — a mismatch.
- Customer confirmed the error from the app's error log and granted permission for the support team to attempt label generation directly. (Comments #3–#5)

### Phase 2: L3 Escalation (2026-06-15)
- Support escalated to senior/L3 team with full context: UPS rates failing with LBS being passed in request despite store units showing KG; orders before and after the affected ones processed successfully.
- Account ID confirmed: `bfbbb21c-18eb-45b4-aa28-1f0e32430b71`. (Comment #11)

### Phase 3: Root Cause Identified, Fix Deferred to Upcoming Release (2026-06-16–2026-06-18)
- Engineering (Veena) confirmed the root cause was identified and requires an app enhancement — not a configuration fix.
- ETA communicated internally and to the customer: fix on or before 7th July 2026.
- Customer was advised to generate labels manually on ups.com in the interim. (Comments #12–#15)

## Open Issues

1. **UPS Label Generation Fails Due to Incorrect Unit of Measurement Sent in API Request for International Orders** — Despite the store unit being set to KG in the MCSL app, the UPS label/rate API request is sending LBS as the unit of measurement for certain orders (68951 and 68952 destined for Germany and Canada). Engineering confirmed a bug — not a customer misconfiguration — and deferred the fix to an upcoming release (ETA on or before 7th July 2026). Customer is blocked from generating labels through the app and must use ups.com manually. Blocked: Customer (workaround available). Severity: High — confirmed engineering bug, ETA set, intermittent (other orders process fine). Area: label-generation. (Comment #11, #12, #15)

## User Story

As a **Director of Operations running a US-based e-commerce store that ships internationally to Europe and Canada on the $29/month plan**,
I want **the app to correctly resolve the unit of measurement from my store settings when building UPS API requests for any destination country**,
so that **I am not forced to leave the app and generate labels manually on ups.com, losing the automation and consolidated workflow I am paying for**.

---

## Acceptance Criteria (Simple)

- [ ] UPS label generation succeeds for international orders when store units are set to KG, with the correct unit of measurement sent in the API request
- [ ] UPS rate requests for destination countries requiring metric units (e.g., Germany, Canada) pass the store-configured unit — not a hardcoded or incorrectly defaulted LBS value
- [ ] Orders 68951 and 68952 for account `bfbbb21c-18eb-45b4-aa28-1f0e32430b71` (localeuropean.myshopify.com) generate labels successfully without manual workaround
- [ ] Orders that previously processed successfully (before and after the affected orders) continue to work without regression
- [ ] Unit of measurement resolution is validated against the UPS-accepted units per destination country to prevent similar mismatches for other country/unit combinations

---

## Acceptance Criteria (Given/When/Then)

### Scenario: International UPS label with store units set to KG
- **Given** a US-based store with units configured as KG in the MCSL app and a UPS carrier connected
- **When** the merchant generates a UPS label for an order destined for Germany or Canada
- **Then** the API request sends KG as the unit of measurement and the label is generated successfully

### Scenario: Mixed unit configurations across orders
- **Given** a store where some orders have product weights in KG and others in LBS
- **When** the merchant generates UPS labels for a batch containing both types
- **Then** each order's API request uses the correct unit of measurement matching the store configuration, and all labels generate without error

### Scenario: Preventing unit mismatch for all carrier-country combinations
- **Given** any supported carrier and any international destination country
- **When** the label generation request is constructed
- **Then** the unit of measurement is resolved from the store's configured units and validated against the carrier's accepted units for that destination, logging a clear error if a genuine incompatibility exists rather than silently defaulting
