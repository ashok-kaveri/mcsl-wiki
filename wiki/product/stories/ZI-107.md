## User Story

As a **Sales Support Specialist at a Netherlands B2B survey-equipment merchant shipping lithium-battery products under IATA PI 967 rules**,
I want **every member of my team to produce identical, compliance-correct UPS labels for the same order, regardless of who clicks Generate**,
so that **I don't have to ask a colleague to re-generate my labels just to get the "PI 967 Section II compliant" marking printed, and we stop falling back to a competitor platform whenever our own labels would ship batteries without the legally required text**.

---

## Acceptance Criteria (Simple)

- [ ] For Emlid (b2b-portal-eu.myshopify.com), a label generated by Jozsef for any multi-product battery order carries the same custom description that a colleague's label would carry
- [ ] The per-user divergence is root-caused and fixed — whether it's Packaging Summary persistence, product-level overrides, user-scoped cached payloads, or permissions
- [ ] A regression test covers "same order, two different StorePep users" and asserts identical label text output
- [ ] When custom descriptions cannot all be printed (e.g., a real carrier-side limit), the UI tells the merchant BEFORE label generation which descriptions will be dropped, not after
- [ ] Jozsef is informed with the root cause, fix, and a verified re-run on one of his real orders — with apology for 36-day SLA breach

---

## Acceptance Criteria (Given/When/Then)

### Scenario 1: Jozsef's own labels now carry the compliance description
- **Given** Jozsef is logged into StorePep and opens a multi-product Shopify order containing a Reach RS3 / RS4 / RX2 battery item with custom description "PI 967 Section II compliant"
- **When** he clicks Generate Label on UPS
- **Then** the printed UPS label contains the custom description for the battery item
- **And** the output is identical to what his colleague would produce from the same order

### Scenario 2: Multi-product custom descriptions handled predictably for any merchant
- **Given** any merchant has a UPS order with 2+ products, each with a custom description
- **When** the label is generated
- **Then** each product's description is either printed on the label or the merchant is warned upfront (with carrier-level explanation) that a description will be dropped
- **And** descriptions are never silently dropped in the request-to-print path

### Scenario 3: Per-user state drift cannot silently change label output
- **Given** two StorePep users in the same store try to generate a label for the same order
- **When** both click Generate Label with identical selections
- **Then** the outbound carrier request payloads and the printed labels are byte-identical for the description field
- **And** a regression test in CI asserts this equivalence so user-scoped state leakage is caught before release
