## Timeline & Key Phases

### Phase 1: Initial FedEx REST Migration Issue (2026-03-17)
- Customer migrated to FedEx REST API as requested by PluginHive
- FedEx rates stopped appearing at checkout after migration
- Support confirmed rates were working, suggested checking automation rules (Comment #2-3)

### Phase 2: Intermittent Rate Failures (2026-03-19 to 2026-03-20)
- Customer reported rates work "sometimes but not all the times"
- Fallback to manual backup prices instead of dynamic FedEx rates
- Customer hypothesis: Too many UPS+FedEx rates causing Shopify timeout — reduced rate count, issue persisted (Comment #6-8)
- Support removed and re-added carrier services, but 2 out of 3 tests still timed out (Comment #12-14)

### Phase 3: L3 Escalation (2026-03-20 to 2026-03-31)
- Escalated to L3: Rates returning successfully in logs but not displaying at checkout (Comment #16)
- Customer reported 2 orders over weekend both used backup rates (Comment #17)
- One week wait with persistent failures — customer escalated urgency (Comment #19, #21)
- **L3 identified root cause**: Token refresh timing issue (Comment #37)
- Fix deployed to QA on 2026-03-31, released 2026-04-01 (Comment #25-28)

### Phase 4: Fix Verification & Ongoing Monitoring (2026-04-02 to 2026-04-16)
- Customer reported fix did NOT resolve issue: 1 out of 5 tests showed correct rates (Comment #29)
- Support verified rates working on multiple test checkouts (Comment #30-31)
- Customer provided evidence: 5 orders since Friday, 4 used backup rates (Comment #32)
- **L3 re-investigation**: Customer had disabled automation rule; L3 re-enabled and monitored (Comment #36)
- **L3 final diagnosis**: "Very specific case where it happens during a token refresh... Will need some dev to fix this" (Comment #37)
- Support confirmed no failures during 3-day monitoring period (Comment #38)

## Open Issues

1. **FedEx REST API rate timeouts during token refresh** — Intermittent checkout rate failures when FedEx OAuth token refresh coincides with rate request. Customer experiences fallback to Shopify backup rates ~80% of the time (4 out of 5 orders). Blocked: customer (unable to provide accurate shipping quotes). Severity: High (revenue impact from incorrect shipping charges, customer frustration after 30-day unresolved case). Area: `carrier-migration`. (Comment #37)

2. **Automation rule stability after REST migration** — Automation rules were disabled by customer (or system), requiring L3 to manually re-enable. Unclear if this is user error or system instability post-migration. Blocked: support (unclear root cause). Severity: Medium (workaround exists but requires manual intervention). Area: `carrier-migration`. (Comment #36)

## Resolved Issues

1. **Initial FedEx REST service registration** — After migration to REST API, carrier services were not properly registered in Shopify. Support removed and re-added services. Partially resolved on 2026-03-20 but revealed underlying token refresh issue. (Comment #12-13)

2. **L3 fix for rate delay (1st attempt)** — L3 deployed fix for "delay in rates" on 2026-04-01, but this did not address the token refresh edge case. (Comment #27-28)

- **Technical setup**: FedEx REST API + UPS, multiple automation rules to filter rates at checkout

## User Story

As a **store owner who was migrated to FedEx REST API and relies on accurate real-time shipping rates to maintain profit margins on international sports equipment shipments**,
I want **FedEx rate requests to succeed consistently at checkout regardless of when OAuth token refresh happens in the background**,
so that **my checkout doesn't randomly fall back to incorrect backup rates that lose me money on 80% of orders, my customers see the actual shipping costs I'll be charged, and I don't have to manually monitor every order to catch pricing errors caused by a timing bug I cannot control or fix myself**.

## Acceptance Criteria (Given/When/Then)

### Scenario: Token refresh does not block rate requests
- **Given** Green Hockey AG's store has FedEx REST API configured
- **When** a customer initiates checkout while the OAuth token is being refreshed
- **Then** the rate request either waits for refresh to complete or uses the expiring token
- **And** FedEx rates display correctly without falling back to Shopify backup rates

### Scenario: High-concurrency checkout reliability
- **Given** multiple customers are checking out simultaneously (weekend order surge)
- **When** FedEx token refresh occurs during active checkout sessions
- **Then** all checkout sessions receive correct FedEx rates
- **And** no customers see backup rates due to token timing issues

### Scenario: Prevent revenue loss from incorrect backup rates
- **Given** store owner previously lost margin on 80% of orders (4 out of 5) due to backup rate fallback
- **When** FedEx REST API token refresh logic is fixed
- **Then** checkout displays accurate FedEx rates for 100% of orders
- **And** store owner no longer manually reviews every order to catch pricing errors
