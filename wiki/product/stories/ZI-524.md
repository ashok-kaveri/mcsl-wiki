## User Story

As a **growing Indian merchant on Amazon Shipping India who bulk-prints labels every morning and has to choose between Amazon Standard and Amazon Express on an order-by-order basis**,
I want **each order's Amazon-quoted estimated delivery days to be visible right in the All Orders and Open Orders grid alongside destination and weight — not buried one click deep in the per-order summary page**,
so that **when I'm scanning 50 orders for the day's manifest, I can see at a glance which ones need Amazon Express to hit a customer's promised delivery date, and I'm not paying for Express on orders that Amazon Standard would deliver in the same window anyway**.

---

## Acceptance Criteria (Simple)

- [ ] In the All Orders grid, every order with an Amazon Shipping India rate has an "Amazon ETA" column showing the estimated delivery days (e.g. "2-3 days" or "3 days") returned by Amazon's rate response, surfaced without requiring the merchant to open the order-summary drawer
- [ ] The same column is available in the Open Orders grid with identical formatting, so the merchant's bulk-label workflow on either screen has parity
- [ ] The Amazon ETA column is sortable and filterable (so Anandhaas can sort by "longest ETA first" to find orders that need an Express upgrade), and persists across grid pagination
- [ ] If Amazon's rate response doesn't include an ETA for an order (carrier outage, address not yet rated), the column shows "—" with a hover tooltip explaining why, rather than a blank cell or a confusing fallback
- [ ] When a merchant toggles other carriers (Delhivery, XpressBees) on the same order alongside Amazon Shipping, the Amazon ETA column continues to show specifically Amazon's quoted ETA — it does not get overwritten by another carrier's promise

---

## Acceptance Criteria (Given/When/Then)

### Scenario: Anandhaas picks Express vs Standard at a glance for tomorrow's batch
- **Given** Anandhaas has 50 orders due for label generation, each rated against Amazon Shipping India in both Standard and Express service classes
- **When** Anandhaas opens the All Orders grid in the morning
- **Then** each row shows the destination, weight, AND Amazon's estimated delivery days in a dedicated column, allowing Anandhaas to sort by ETA descending and pick the 5 longest-ETA orders for Amazon Express upgrade before triggering the bulk label run

### Scenario: ETA column works across MCSL's full grid surface, not just one screen
- **Given** the merchant has the ETA enhancement enabled and is filtering down to a specific pincode region
- **When** the merchant switches between All Orders and Open Orders views and applies the same pincode filter on both
- **Then** the Amazon ETA column appears in both views with identical rendering, the filter and sort state are preserved when switching, and grid pagination does not reset the ETA values

### Scenario: ETA gracefully degrades when Amazon doesn't return one
- **Given** an order whose destination Amazon Shipping cannot currently rate (e.g. coverage gap, weight exceeds Amazon limits, or transient API failure)
- **When** the merchant looks at that row in the order grid
- **Then** the Amazon ETA column shows "—" with a tooltip explaining the reason ("Amazon could not quote rates for this destination"), and the merchant can still sort/filter the column without those rows polluting the sort order
