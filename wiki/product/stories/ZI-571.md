- **Product**: WooCommerce Shipping Services (WSS) Plugin

## Timeline & Key Phases

### Phase 1: Initial Report & Webhook Refresh (May 12-13)
- Customer reports WooCommerce processing orders not auto-importing into WSS plugin; manual import requires 2+ clicks to trigger
- Support identifies customer context: US store, UPS carrier, $19 plan, 87 labels in May, active since Oct 2024
- Support discovers webhooks exist but may need refresh; refreshes all webhooks
- Support confirms new orders (12083, 12084) now imported successfully after refresh
- Customer provides WP admin access (credentials shared)

### Phase 2: Repeated Failures & Product Issue (May 13-15)
- May 13: Customer reports 2 additional orders (12062, 12082) failed to import; support discovers products duplicated in plugin database and deletes/reimports all products fresh; orders then import
- May 14: New orders continue to fail auto-import; support requests customer wait for next occurrence without manually importing
- May 14-15: Support escalates to L3 team with "Need Assistance from L3" flag
- L3 (Veena) enables app logs on store to capture webhook failures in real-time
- Customer reports orders 12105, 12108, 12110 placed 4+ hours ago not in plugin; support promises investigation by Monday

### Phase 3: Access Negotiation & Root Cause Discovery (May 19-22)
- May 19: L3 requests sample order and FTP access to investigate
- May 20: Customer offers WP File Manager plugin access; support declines due to risk
- May 22: Customer provides FTP credentials verified as working
- **May 22: ROOT CAUSE #1 IDENTIFIED** — Webhook was tied to non-admin user account lacking `woocommerce_rest_cannot_view` permission, causing 401 Unauthorized errors. Support deleted existing webhook and instructed customer to recreate with admin account
- May 22: Support requests clarification on exact webhook setup steps (Order Created, Order Updated, delivery URL)

### Phase 4: Webhook Reset & Persistence (May 25-27)
- May 25: L3 clears all webhooks from plugin side; customer performs ReSync via admin account

- May 25: Customer asks: "Will future orders auto-import now?" Support confirms yes, should appear within 1-2 minutes
- **May 26: ISSUE PERSISTS** — Customer reports orders 12274-12277 placed but still not auto-importing despite webhook fix
- May 26: L3 checks store and finds Website Data Access Test (query by order/product ID) returns no data, while webhooks are active, Store JSON working, and API test passing
- May 27: L3 escalates internally: "We have identified the issue with orders not getting imported through webhooks. We are working on the fix."
- May 27: Ticket moved to `dev_needed` status; customer acknowledged; support advised to manually import pending orders

## Open Issues

1. **Orders Not Auto-Importing — Root Cause #2 (Critical)** — Initial webhook permission issue (fixed May 22, root cause #1) did not resolve the problem. L3 identified secondary issue May 26: Website Data Access Test failing — orders placed but no details returned when plugin queries by order/product ID, despite all diagnostic tests passing (webhooks active, Store JSON working, API response tester working). This suggests data retrieval layer or permission issue beyond webhook delivery. Blocked: L3 development team awaiting permanent fix. Severity: Critical for customer (shipping backlog accumulation, manual import labor every 4-6 hours). Area: order-management, onboarding. (Comments #55-56)

2. **Manual Import Button Requires Multiple Clicks** — Customer reported initial issue: clicking Import Orders button requires 2+ refresh attempts to trigger import batch. Not root-caused; support attributed to "plugin sync time" but never verified if resolved. Likely related to root cause #2 (data access timing). Severity: Minor usability issue. Area: order-management. (Comment #1)

## Resolved Issues

1. **Webhook Permission Denied (401 Unauthorized)** — L3 diagnosed webhook tied to non-admin user account without `woocommerce_rest_cannot_view` REST API permission. Error: "woocommerce_rest_cannot_view 401 Unauthorized". Webhook deleted; customer instructed to recreate with administrator account. Resolved May 22. (Comments #40-41)

2. **Duplicate Products Blocking Import** — Orders 12062 and 12082 contained products that were duplicated in plugin database, preventing import. Support deleted all products and reimported fresh. Orders then imported successfully. Resolved May 13. (Comment #17)

## User Story

As a **WooCommerce store operator shipping ~87 packages/month on a $19 plan, relying on a dev liaison to manage plugin integrations across time zones**,
I want **orders to flow automatically from WooCommerce into the shipping plugin without manual intervention, even when the underlying data-access layer encounters query failures against WooCommerce REST endpoints**,
so that **my shipping team stops losing 4-6 hours every day manually importing orders, my fulfillment backlog stops growing while we wait for a fix that has dragged on for over five weeks, and my client does not abandon the platform after investing 20 months building their workflow around it**.

---

## Acceptance Criteria (Simple)

- [ ] WooCommerce orders with status "Processing" are auto-imported into the WSS plugin within 2 minutes of order placement, verified on the American Natural Premium store (americannaturalpremium.com)
- [ ] The Website Data Access Test returns order and product details when queried by order ID or product ID, matching the data available via the WooCommerce REST API
- [ ] Webhook-triggered order imports succeed regardless of whether the webhook was originally created by an admin or non-admin user account, with automatic permission validation during webhook setup
- [ ] When the data-access layer fails to retrieve order details from WooCommerce, the plugin logs the failure with the specific WooCommerce REST API error code and retries with exponential backoff before surfacing an alert
- [ ] A dashboard health indicator flags when the data-access query path diverges from the direct API response path, preventing silent import failures from persisting undetected for days

---

## Acceptance Criteria (Given/When/Then)

### Scenario: Auto-import succeeds after data-access layer fix is deployed
- **Given** the WSS plugin is installed on a WooCommerce store with active webhooks (order.created, order.updated) and the data-access layer fix has been deployed
- **When** a customer places a new order that transitions to "Processing" status in WooCommerce
- **Then** the order appears in the WSS plugin order list within 2 minutes, with all line items, shipping address, and product details fully populated without any manual import action

### Scenario: Data-access query returns results consistent with direct REST API
- **Given** a WooCommerce store has orders and products accessible via the REST API (verified by the API response tester returning valid JSON with order details)
- **When** the WSS plugin executes the Website Data Access Test by querying a specific order ID or product ID through the plugin's internal data retrieval path
- **Then** the test returns the same order/product data that the direct WooCommerce REST API call returns, with no empty or null responses for records that exist in WooCommerce

### Scenario: Transient data-access failures are retried with visibility
- **Given** the WooCommerce REST API is temporarily unreachable or returns a 5xx error during a webhook-triggered import attempt on a store with active orders
- **When** the plugin's data-access layer encounters the retrieval failure while processing an incoming order.created webhook payload
- **Then** the plugin retries the data retrieval up to 3 times with exponential backoff, logs each attempt with the HTTP status code and order ID, and surfaces a persistent dashboard alert if all retries are exhausted
