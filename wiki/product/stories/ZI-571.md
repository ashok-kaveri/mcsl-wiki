## Open Issues

1. **Orders Not Auto-Importing — Root Cause #2 (Critical)** — Initial webhook permission issue (fixed May 22, root cause #1) did not resolve the problem. L3 identified secondary issue May 26: Website Data Access Test failing — orders placed but no details returned when plugin queries by order/product ID, despite all diagnostic tests passing (webhooks active, Store JSON working, API response tester working). This suggests data retrieval layer or permission issue beyond webhook delivery. Blocked: L3 development team awaiting permanent fix. Severity: Critical for customer (shipping backlog accumulation, manual import labor every 4-6 hours). Area: order-management, onboarding. (Comments #55-56)

2. **Manual Import Button Requires Multiple Clicks** — Customer reported initial issue: clicking Import Orders button requires 2+ refresh attempts to trigger import batch. Not root-caused; support attributed to "plugin sync time" but never verified if resolved. Likely related to root cause #2 (data access timing). Severity: Minor usability issue. Area: order-management. (Comment #1)

## Resolved Issues

1. **Webhook Permission Denied (401 Unauthorized)** — L3 diagnosed webhook tied to non-admin user account without `woocommerce_rest_cannot_view` REST API permission. Error: "woocommerce_rest_cannot_view 401 Unauthorized". Webhook deleted; customer instructed to recreate with administrator account. Resolved May 22. (Comments #40-41)

2. **Duplicate Products Blocking Import** — Orders 12062 and 12082 contained products that were duplicated in plugin database, preventing import. Support deleted all products and reimported fresh. Orders then imported successfully. Resolved May 13. (Comment #17)

## User Story

As a **WooCommerce merchant whose orders stopped auto-importing 19 days ago — forcing my team to manually trigger imports every few hours to keep fulfillment running — because the webhook was tied to a non-admin user account that returns 401 Unauthorized on order data requests**,
I want **the webhook order import pipeline to validate API user permissions during setup and automatically recover when the underlying data retrieval layer fails, even after the webhook itself is correctly configured**,
so that **orders flow into the plugin automatically like they did before, my team stops babysitting manual imports every 4-6 hours, and the next time a permission issue blocks imports the system tells me exactly what's wrong instead of silently failing for weeks**.

---

## Acceptance Criteria (Simple)

- [ ] Webhook setup validates that the connected WooCommerce user has admin-level order data access before saving
- [ ] Auto-import processes orders reliably when webhook fires — no silent 401 failures
- [ ] When the data retrieval layer fails (Website Data Access Test), the plugin surfaces a clear error in the dashboard: "Order import blocked: API user lacks order read permissions"
- [ ] Jason Ballantyne's store (American Natural Premium) auto-imports orders without manual intervention after the fix
- [ ] Manual import fallback works with a single click (not multiple clicks required)

## Acceptance Criteria (Given/When/Then)

### Scenario: Webhook with valid admin user
- **Given** a WooCommerce store with webhook configured using an admin user account
- **When** a new order is placed in WooCommerce
- **Then** the order auto-imports into WSS within 60 seconds
- **And** the import log shows "Webhook received → Order imported successfully"

### Scenario: Non-admin user blocked at setup
- **Given** a merchant tries to configure the webhook with a non-admin user account
- **When** they save the webhook configuration
- **Then** the system shows "Error: This user lacks order data permissions. Please use an admin account."
- **And** the webhook is not saved until a valid admin user is provided

### Scenario: Data retrieval failure surfaced clearly
- **Given** the webhook is correctly configured but the underlying data access test fails
- **When** an order webhook fires and the data retrieval returns empty
- **Then** the dashboard shows a persistent alert: "Orders not importing — Website Data Access Test failing"
- **And** the alert includes a "Recheck Permissions" button to re-validate
