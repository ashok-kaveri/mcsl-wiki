## User Story

As a **merchant who installed the plugin expecting it to work reliably**,
I want **the plugin to load correctly every time without cryptic "isActive:false" errors that I can't diagnose myself**,
so that **I don't have to uninstall and lose my shipping automation when a technical flag breaks functionality, and I don't need to open support tickets just to understand why basic plugin loading failed**.

---

## Acceptance Criteria (Simple)

- [ ] Plugin loads successfully even when isActive flag is temporarily false due to subscription sync delays
- [ ] Real-time API subscription check serves as fallback when localStorage flags are stale or out of sync
- [ ] emmetts.com.au customer (#386476) can access plugin dashboard and generate labels without loading errors
- [ ] Detailed logging captures flag mismatches (cached value vs API response) for debugging without requiring support tickets
- [ ] Automated alerts notify development team when isActive=false loading failures occur to prevent customer churn

---

## Acceptance Criteria (Given/When/Then)

### Scenario: Plugin loads correctly even when isActive flag is temporarily false
- **Given** emmetts.com.au (customer #386476) has the plugin installed but isActive flag is set to false due to subscription sync delay
- **When** the merchant accesses the plugin dashboard or generates a label
- **Then** the plugin initializes successfully without loading errors
- **And** the system checks subscription status via API as fallback if localStorage flag is stale
- **And** the merchant can continue using the plugin while subscription status reconciles

### Scenario: All subscription status flags are gracefully handled with fallbacks
- **Given** any subscription status flag (isActive, isTrial, isPaid) becomes out of sync with actual subscription state
- **When** the plugin attempts to load or perform operations
- **Then** the plugin uses real-time API subscription check as source of truth (not cached flags alone)
- **And** stale flags are updated automatically based on API response
- **And** no functionality is blocked due to flag desynchronization

### Scenario: Flag validation logging helps diagnose and prevent future issues
- **Given** a subscription flag mismatch occurs
- **When** the plugin detects the discrepancy between cached flag and API status
- **Then** detailed logging captures the mismatch (flag value, API response, timestamp, user store)
- **And** error logs are accessible for debugging without requiring merchant to open a ticket
- **And** development team receives automated alerts for isActive=false loading failures to proactively fix before churn
