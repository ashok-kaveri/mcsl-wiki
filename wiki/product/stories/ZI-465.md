## User Story

As an **Australia-based merchant with daily packing/dispatch operations and same-day shipping deadlines**,
I want **to generate CouriersPlease labels directly through the app with clear error messages when failures occur**,
so that **my team doesn't waste time manually printing labels through the carrier's portal, losing money on increased operational costs while orders sit waiting for dispatch**.

## Acceptance Criteria (Simple)

- [ ] Hispanic Pantry store (hispanic-pantry.myshopify.com) can successfully generate CouriersPlease labels for orders HP#110710926 and HP#110710886
- [ ] Error messages for CouriersPlease label failures are specific enough to identify whether the issue is app-side or carrier API-side
- [ ] When CouriersPlease returns a 403 error, the app displays the actual carrier response message instead of "Unidentified Error"
- [ ] Label failure errors include actionable next steps for the merchant (e.g., "Contact courier support" vs "Check order details")
- [ ] Similar label failures for any Australian courier (CouriersPlease, MPB, Australia Post, etc.) provide diagnostic error messages instead of generic codes

## Acceptance Criteria (Given/When/Then)

### Scenario: Retry failed CouriersPlease labels
- **Given** Hispanic Pantry has orders HP#110710926 and HP#110710886 with previous "403 Forbidden" and "Unidentified Error" failures
- **When** merchant retries label generation for these orders
- **Then** labels generate successfully OR display specific error message indicating root cause (e.g., "CouriersPlease API returned 403: Invalid credentials" or "CouriersPlease API returned 403: Service unavailable for this postal code")

### Scenario: Clear error diagnostics for all CouriersPlease label failures
- **Given** a merchant attempts to generate a CouriersPlease label for any order
- **When** the label generation fails due to a carrier API error
- **Then** the error message includes: (1) HTTP status code, (2) carrier's error message verbatim, (3) whether the issue is app-side or carrier-side, (4) recommended action ("Retry", "Contact support", "Update order details")
- **And** the error is logged to L3 with full request/response details for troubleshooting

### Scenario: Prevent generic "Unidentified Error" for any carrier
- **Given** any carrier API (CouriersPlease, USPS, FedEx, UPS, etc.) returns a structured error response
- **When** label generation fails
- **Then** the app displays the carrier's specific error message instead of "Unidentified Error"
- **And** ambiguous errors (403, 500, timeout) trigger automatic L3 investigation logging with full API payload
