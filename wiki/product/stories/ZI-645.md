## User Story

As a **high-volume WooCommerce shipper using CanPar through the WSS plugin (~10,122 labels/month)**,
I want **the CanPar service display names in the plugin to match the official CanPar service names I see on my CanPar account and shipping documents**,
so that **my team can confidently select the correct service without cross-referencing external documentation, reducing mislabelled shipments and customer complaints**.

---

## Acceptance Criteria (Simple)

- [ ] All CanPar service codes returned by the API are mapped to their current official CanPar display names
- [ ] The renamed services appear consistently in rate shopping results, automation rule dropdowns, and label generation confirmations
- [ ] Existing automation rules referencing old CanPar service names continue to function without manual reconfiguration
- [ ] The service renaming does not affect CanPar rate calculation or label generation payloads
- [ ] A CanPar service name mapping is maintainable via configuration rather than hardcoded strings

---

## Acceptance Criteria (Given/When/Then)

### Scenario: Renamed services appear in rate shopping
- **Given** a WooCommerce store with an active CanPar carrier account connected to the WSS plugin
- **When** the merchant fetches shipping rates for an order destined within Canada
- **Then** the CanPar services listed in the rate results display the updated official CanPar service names (e.g., "CanPar Express" instead of any legacy or internal code name)

### Scenario: Automation rules with old service names still trigger correctly
- **Given** the merchant has existing automation rules configured with the previous CanPar service display names
- **When** an order matching those automation rule conditions is processed after the service renaming deployment
- **Then** the automation rules match and apply correctly, selecting the intended CanPar service without requiring the merchant to edit the rule

### Scenario: Label generation uses correct service after renaming
- **Given** the merchant selects a CanPar service by its new display name for an order
- **When** the label is generated via the CanPar API
- **Then** the API request sends the correct underlying service code and the generated label reflects the correct service, with the new display name shown in the plugin's shipment confirmation
