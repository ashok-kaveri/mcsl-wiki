## User Story

As a **PluginHive product team expanding carrier coverage into European markets where GLS has strong presence (Czech Republic, Ireland, Spain) and where API credentials and partnership terms are already in hand**,
I want **a phased GLS shipping API integration starting with ShipIT for label generation across priority country clusters, building on the Track & Trace work Manu already completed**,
so that **we capture underserved European markets before competitors fill the gap, honor our partnership commitment to GLS who are waiting on volume before deepening the relationship, and stop leaving merchant revenue on the table in regions where GLS is the dominant domestic carrier**.

## Acceptance Criteria (Given/When/Then)

### Scenario: GLS ShipIT label generation for supported country cluster
- **Given** a merchant has connected GLS ShipIT credentials for a supported country cluster (e.g., Czech Republic)
- **When** the merchant generates a shipping label for an order with a GLS-eligible destination
- **Then** the system creates a valid GLS shipment via ShipIT API and returns a printable label with tracking number linked to the existing Track & Trace integration

### Scenario: Country cluster routing to correct GLS API
- **Given** GLS requires different APIs for different country clusters (global ShipIT vs. national APIs like Spain)
- **When** a merchant ships to a destination covered by a national API rather than ShipIT
- **Then** the system either routes the request to the correct regional API endpoint or clearly indicates the destination is not yet supported with a user-facing message

### Scenario: Phased rollout does not break existing Track & Trace
- **Given** GLS Track & Trace API integration is already live and functional
- **When** the ShipIT shipping API integration is deployed alongside it
- **Then** existing tracking functionality continues to work without regression, and new shipments created via ShipIT are automatically trackable through the same Track & Trace pipeline
