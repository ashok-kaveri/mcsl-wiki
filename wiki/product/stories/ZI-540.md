## User Story

As a **PluginHive product owner who has just been handed a multi-stream NZ Post roadmap by our partnerships team — SOv2 API parity, label enhancements (logo + 1D barcode for returns), and a partner expecting a feasibility-and-ETA reply within weeks**,
I want **MCSL's NZ Post integration upgraded to the SOv2 Shipping API (consignment pricing, Signature Required add-on, Fliway oversize handling, Express service) and the Parcel Label API enhancements (platform logo, merchant 1D barcode) shipped together in the Q4 release window, with a single scoping document going back to NZ Post that covers feasibility, dependencies, effort, and ETA for all three streams**,
so that **NZ Post merchants on MCSL stop falling behind on rate accuracy and label features as NZ Post evolves its API, our partnerships team can keep one credible conversation with NZ Post (instead of three half-answers), and we don't lose the SOv2 release window that NZ Post is currently giving every shipping platform — including the ones that compete with us**.

---

## Acceptance Criteria (Simple)

- [ ] MCSL's NZ Post integration calls the SOv2 Shipping Options endpoint for rate quotes (replacing or augmenting the prior endpoint), and the response correctly surfaces consignment-level pricing, the Signature Required add-on (for approved-sender NZ Post accounts), Fliway oversize routing, and Express service options inside the MCSL rate panel
- [ ] NZ Post labels generated by MCSL include the PluginHive shipping-platform logo per NZ Post's new label spec, and accept a merchant-supplied 1D barcode for the returns/inbound workflow (with the barcode rendered correctly per NZ Post's barcode requirements)
- [ ] A scoping document goes back to NZ Post within one PluginHive sprint covering: per-stream feasibility, MCSL-side dependencies (e.g. new credential fields, account-type checks), effort estimate, target release window — and is signed off by Abhilash before being sent
- [ ] A live merchant testing the SOv2 integration sees Signature Required as a per-order add-on selection (not a global toggle), and the resulting label charges back the correct rate including any add-on premium
- [ ] All three streams ship behind a feature flag scoped to NZ Post-connected merchants, so we can dark-launch SOv2 to a pilot subset before flipping it on for the full NZ Post merchant population

---

## Acceptance Criteria (Given/When/Then)

### Scenario: NZ Post merchant on MCSL sees Express + Signature Required available against a real order
- **Given** an approved-sender NZ Post merchant connected to MCSL via the SOv2-enabled integration, generating a rate request for a domestic NZ shipment
- **When** they open the order's rate panel
- **Then** the rate response includes both Standard and Express NZ Post services, the Signature Required add-on is selectable per service, and the rate shown reflects consignment-level pricing for the actual package dimensions/weight — not the simpler unit rate from the older endpoint

### Scenario: Returns workflow uses merchant-supplied 1D barcode on NZ Post label
- **Given** an NZ Post merchant who supplies a 1D barcode value for a returns shipment (e.g. their RMA number)
- **When** MCSL generates the NZ Post return label via the upgraded Parcel Label API
- **Then** the label includes the merchant's 1D barcode rendered correctly per NZ Post specs, the PluginHive platform logo appears in the platform-logo slot, and a scan of the printed label resolves to the merchant's RMA value in their inbound system

### Scenario: Partnership signals back to NZ Post stay coherent through the rollout
- **Given** Abhilash committed PluginHive to a feasibility + ETA reply across all three NZ Post streams
- **When** engineering completes the scoping pass
- **Then** a single document goes back to NZ Post listing the three streams with per-stream feasibility, dependencies, effort, and a unified target ETA tied to PluginHive's Q4 release — NZ Post does not receive three different timelines from three different threads, and the document is archived in MCSL's product/decisions folder for future reference
