## User Story

As a **high-volume Shopify merchant (Batoony Rocks) whose warehouse staff pick orders by reading the Order ID off each Amazon label**,
I want **the Order ID printed large enough that a picker can't confuse a 6 for an 8**,
so that **I stop sending wrong parcels to customers, stop eating the cost of misdeliveries, and don't have to switch to an alternative shipping app just to get a legible label**.

---

## Acceptance Criteria (Simple)

- [ ] Batoony Rocks can read the Invoice ID / Order ID on every Amazon label their warehouse prints without ambiguity between similar digits.
- [ ] PluginHive has confirmed in writing from the Amazon Shipping API team whether font/layout on Amazon labels can be customised.
- [ ] If Amazon API does not allow label changes, PluginHive offers a supplementary slip or enlarged Order ID reference (e.g. packing slip add-on, large prefix sticker) that prints alongside the label.
- [ ] A PluginHive-side option (e.g. Order ID prefix `#` + bold) is available to any MCSL merchant using Amazon labels.
- [ ] L3 ticket is closed with a customer-visible decision documented, not just an internal Slack thread.

---

## Acceptance Criteria (Given/When/Then)

### Scenario: Batoony Rocks' warehouse picks reliably
- **Given** Batoony Rocks prints an Amazon label for a Shopify order via MCSL
- **When** a warehouse picker looks at the label
- **Then** the Order ID is legible at arm's length with no 6/8/0/O/1/l confusion
- **And** the customer confirms misdelivery incidents have stopped

### Scenario: Any Amazon-label customer has an escape hatch
- **Given** any MCSL merchant using the Amazon Shipping integration raises readability concerns
- **When** they open their label settings
- **Then** they can enable a supplementary Order ID slip or large-text overlay
- **And** the option is documented in the app help

### Scenario: Prevent churn from API-limit replies
- **Given** a customer raises a label-quality concern attributed to a carrier API limit
- **When** support responds
- **Then** the reply includes either a PluginHive-side workaround OR a recorded outreach to the carrier API team with an expected response date
- **And** support does NOT close the ticket with "the API doesn't allow it" as the final answer
