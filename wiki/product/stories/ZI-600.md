## Open Issues

1. **ConsigneeTypeValue (B2B/B2C Indicator) Missing from UPS EU Shipments** — UPS Ship API field `ShipmentRequest/Shipment/GlobalTaxInformation/ConsigneeTypeValue` must be populated for EU-bound shipments by July 1, 2026. Although UPS will not enforce it at the API level to avoid hard failures, missing data will cause customs holds and return-to-shipper at partner's cost. StorePep must capture and transmit this field. Blocked: engineering. Severity: deadline-driven compliance risk — customs delays and RTS charges if missed; July 1 applies based on customs arrival date, not label manifest date. Area: international. (Comment #1, #3)

2. **IOSS Data Fields Need Update for UPS EU Shipments** — As of July 1, IOSS Name and Address fields are no longer required on commercial invoices; only the IOSS Number is required. Current StorePep UPS IOSS implementation likely transmits name/address fields that are now deprecated. Must verify and strip deprecated fields to stay compliant. Blocked: engineering. Severity: moderate — incorrect IOSS data may cause invoice rejection or customs friction. Area: international. (Comment #1, #3)

3. **Product ID Field Capture Needed in Preparation for November 1, 2026 Enforcement** — UPS has postponed Product ID requirements until November 1, 2026, but instructs partners to begin capture now. StorePep does not currently collect Product IDs at the line-item level for UPS EU shipments. Implementation should begin before the November deadline. Blocked: engineering (planning phase). Severity: low urgency now, high urgency by October. Area: international. (Comment #1, #3)

## User Story

As a **Shopify merchant shipping B2B and B2C orders to EU customers through UPS**,
I want **my shipments to automatically include the correct consignee type classification and compliant customs data so EU customs processes them without manual intervention**,
so that **my packages don't get held at customs or returned to me — costing me reshipping fees, refund requests, and customers who never order internationally from me again**.

---

## Acceptance Criteria (Simple)

- [ ] UPS EU-bound shipments include `ConsigneeTypeValue` in the `GlobalTaxInformation` block of the Ship API request, set to the correct B2B or B2C indicator based on the order's recipient type
- [ ] IOSS data transmitted to UPS contains only the IOSS Number — deprecated Name and Address fields are stripped from the commercial invoice payload
- [ ] Existing UPS EU shipment flows (label generation, manifest, rate shopping) continue to function without regression after the compliance fields are added
- [ ] Merchants shipping to EU destinations before July 1, 2026 see no change in workflow; the compliance data is populated automatically without additional merchant input
- [ ] A validation or fallback mechanism exists so that if consignee type cannot be determined, the shipment either defaults safely or surfaces a clear error rather than silently omitting the field

## Acceptance Criteria (Given/When/Then)

### Scenario: B2C shipment to EU includes ConsigneeTypeValue
- **Given** a merchant creates a UPS shipment to an EU destination address where the recipient is an individual consumer (no company name)
- **When** the label is generated and the Ship API request is constructed
- **Then** the request body includes `ShipmentRequest.Shipment.GlobalTaxInformation.ConsigneeTypeValue` set to the B2C indicator value, and the label generates successfully

### Scenario: B2B shipment to EU includes ConsigneeTypeValue
- **Given** a merchant creates a UPS shipment to an EU destination address where the recipient is a business (company name present)
- **When** the label is generated and the Ship API request is constructed
- **Then** the request body includes `ShipmentRequest.Shipment.GlobalTaxInformation.ConsigneeTypeValue` set to the B2B indicator value, and the label generates successfully

### Scenario: IOSS fields are compliant with July 1 requirements
- **Given** a merchant has configured their IOSS Number in StorePep and generates a UPS EU shipment
- **When** the Ship API request is constructed with IOSS data
- **Then** only the IOSS Number is included in the payload — IOSS Name and IOSS Address fields are not transmitted

### Scenario: Non-EU UPS shipments are unaffected
- **Given** a merchant creates a UPS shipment to a non-EU destination (e.g., US, Canada, Australia)
- **When** the label is generated
- **Then** the `ConsigneeTypeValue` field is not added to the request, and the shipment processes identically to pre-change behavior
