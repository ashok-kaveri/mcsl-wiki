## Open Issues

1. **Postnord automation rule "Set Default Service Point" overrides customer-selected service point** — When both (a) the "Set Default Service Point for PostNord" automation rule is active AND (b) the customer has chosen a service point at checkout, the rule wins and assigns the *closest* SP instead of honoring the customer's choice. The rule needs to act only as a fallback for orders without a customer selection — i.e. customer-selected SP should take precedence. Confirmed reproducible on order #33295 (customer chose "Lähiboksi Kauppakeskus Lauttis 1", app generated label with "K-market Kasinonranta"). Workaround offered: bypass the app and generate label on PostNord's website. Blocked: engineering. Severity: `dev_needed` (release 380 targeted). Area: `label-generation`. (Comments #11, #13, #14)

## Resolved Issues

1. **Removing the automation rule broke the no-customer-choice fallback (initial misdiagnosis)** — Following the original L1 advice to delete the "Set Default Service Point" action caused every order without a customer-selected SP to fail label generation. Reverted on 2026-05-08 by re-enabling the rule, restoring the closest-SP fallback. Confirmed this was a side-effect of the advice, not a separate bug — the underlying issue is the rule's priority order (it should defer to customer-selected SP, not override it). (Comments #4, #7, #8, #9, #10)

## User Story

As a **Finnish merchant who relies on PostNord pickup points for every shipment and lets buyers pick their preferred locker or shop at checkout**, I want **the buyer's chosen service point to be the one printed on the label every single time — and the "Set Default Service Point" automation rule to only kick in when the buyer actually skipped the picker**, so that **packages arrive at the locker the buyer can actually walk to, my refund/return queue stops filling with "this was sent to the wrong shop" complaints, and I am not forced to choose between (a) generating labels manually on PostNord's website and (b) leaving the automation rule on and shipping to the wrong location**.

---

## Acceptance Criteria (Simple)

- [ ] When an order has a customer-selected PostNord service point at checkout, the label is generated using that exact service point regardless of whether the "Set Default Service Point" automation rule is active
- [ ] When an order has no customer-selected service point, the "Set Default Service Point" rule continues to auto-pick the nearest PostNord SP (the existing fallback behaviour must not regress — see ZI-misdiagnosis phase 2 of #387640)
- [ ] Maustemesta's order #33295 — when re-run — produces a label for "PostNord Lähiboksi Kauppakeskus Lauttis 1" (customer's choice), not "PostNord K-market Kasinonranta" (the nearest one)
- [ ] The fix is verified end-to-end against both order paths (with-selection and without-selection) before release, so we never again offer the merchant the binary "disable the rule and break label gen" / "keep it on and break customer choice"
- [ ] Release notes for MCSL 380 explicitly call out the priority change ("customer-selected SP always wins over default-SP rule") so other PostNord merchants understand the new behaviour without a ticket

## Acceptance Criteria (Given/When/Then)

### Scenario: Customer-selected PostNord service point is honoured at label generation

- **Given** an order on a PostNord-only store with the "Set Default Service Point for PostNord" automation rule active
- **And** the buyer selected a specific service point ("PostNord Lähiboksi Kauppakeskus Lauttis 1") at Shopify checkout
- **When** the merchant generates the label via the app
- **Then** the label is generated for the buyer's selected service point ("Lähiboksi Kauppakeskus Lauttis 1")
- **And** the automation rule does NOT override that selection with the nearest SP

### Scenario: No-customer-choice fallback continues to work

- **Given** the same store with the same automation rule active
- **And** an order where the buyer skipped the PostNord service-point picker
- **When** the merchant generates the label
- **Then** the app auto-selects the nearest PostNord service point and the label is generated successfully
- **And** the merchant is NOT forced to disable the automation rule to avoid label-generation failures (the regression reported in comments #7–#9 stays fixed)

### Scenario: Prevention — the rule's priority order is explicit and testable

- **Given** a future merchant configures any "Set Default" automation action (PostNord, or future carriers that gain pickup-point selection)
- **When** an order arrives carrying a customer-selected value for the same field
- **Then** the customer-selected value always wins; the "Set Default" action behaves purely as a fallback for missing values
- **And** this precedence is covered by an automated regression test against an order fixture with a customer-chosen service point, so the next refactor cannot silently invert the rule
