## User Story

As an **Australian agency developer who has already collected payment from my end client for the WSS plugin and is now five weeks into a go-live that won't close**, I want **postcode-driven shipping zones to actually drive carrier rates at checkout the moment I configure them — without me filing a video and waiting hours for L3 to manually rewrite the same zone for a second time in five days**, so that **my client stops asking me when their store will accept Sydney orders, I stop having to apologise for a plugin I recommended, and we don't burn another support cycle on a configuration step that should be self-serve.**

---

## Acceptance Criteria (Simple)

- [ ] Postcodes added to a WooCommerce Shipping Services shipping zone via the standard plugin UI are matched at checkout without an L3 agent having to manually re-enter them in the backend.
- [ ] When a postcode is present in the zone but no rates appear at checkout, the plugin surfaces the specific reason in the rate-debug log (zone unmatched / rule not attached / "Match All Conditions" blocking / packaging dimension mismatch) — not a silent "no rates available" message.
- [ ] About Bunting's TNT Australia + AusPost setup at `aboutbunting.sitestaging.com.au` returns rates for every Sydney postcode the agency has loaded (2000--2234, 2055--2060, 2145--2155, 2760--2770, and the rural sample 3460) without further L3 intervention.
- [ ] The agency dev (Jarrod) can validate zone-to-rate matching for an arbitrary postcode via a self-serve diagnostic in the plugin — without filing a ticket or recording a Loom video.
- [ ] A regression test covers the recurring pattern from this ticket — zone created in UI, postcodes pasted in bulk, "Match All Conditions" defaulted off, rate automation rule auto-linked to the new zone — so that a future merchant doesn't reproduce the same L3 escalation twice in five days.

## Acceptance Criteria (Given/When/Then)

### Scenario: Agency dev re-runs the failed Sydney postcode test after the fix

- **Given** the About Bunting store has a TNT Australia shipping zone containing Sydney postcodes `2000, 2055, 2145, 2250, 2251, 2760` pasted via bulk-paste from Excel, and the TNT rate automation rule is attached to that zone with "Match All Conditions" disabled
- **When** Jarrod adds a 5-metre bunting product to the cart and proceeds to checkout with shipping address `2250 NSW`
- **Then** TNT Road Express and TNT Overnight Express both appear as rate options at checkout in the first attempt — not on the second, not after a cache clear, not after an incognito retry
- **And** the same outcome holds for every other Sydney postcode loaded in the zone (2000, 2055, 2145, 2251, 2760) without any agent reconfiguration

### Scenario: Diagnose why a configured postcode is not returning a rate

- **Given** a WSS merchant has added postcodes to a shipping zone via the plugin UI but no rates appear at checkout for those postcodes
- **When** the merchant opens the rate-debug log for that checkout attempt (or runs a diagnostic against a single postcode from the plugin admin)
- **Then** the log states the specific reason in plain language — "zone matched but rate automation rule not attached", "Match All Conditions blocked rule evaluation", "packaging method requires product weight that is missing", or "carrier API returned no service for this lane"
- **And** the message includes a one-click jump or doc link to the exact setting that needs to change

### Scenario: Prevent the same L3 escalation pattern on the next WSS go-live

- **Given** a new WooCommerce Shipping Services merchant creates their first shipping zone via the plugin's zone wizard
- **When** they paste a list of postcodes and save the zone
- **Then** the wizard automatically (a) defaults "Match All Conditions" to disabled, (b) prompts the merchant to link a rate automation rule to the new zone before saving, and (c) runs a smoke test against one postcode from the pasted list and surfaces the result inline
- **And** if any of those defaults are not respected, the zone cannot be saved without the merchant explicitly overriding each one — so support is no longer required to discover the same misconfiguration via a Loom video review
