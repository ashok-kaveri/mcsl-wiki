## User Story

As a **Dubai-based merchant who has been forced into FedEx's REST migration and is relying on the SOAP integration to keep my Estimated Delivery numbers honest at checkout**, I want **every per-carrier setting I configured on FedEx SOAP — including "Add Buffer Days to Estimated Delivery" — to come over to FedEx REST automatically as part of the migration**, so that **the day I switch to REST is the day I switch to REST, not the day my checkout starts quoting delivery dates I cannot meet, and I am not stranded between an SOAP integration that's being retired and a REST integration that's silently missing features I depend on**.

---

## Acceptance Criteria (Simple)

- [ ] FedEx REST's Carrier Settings → Other Details exposes the same "Add Buffer Days to Estimated Delivery (upto 10 days)" toggle that FedEx SOAP exposes today
- [ ] Migrating a store from FedEx SOAP → FedEx REST preserves the merchant's existing buffer-days value (no manual re-config required post-migration)
- [ ] thedeensquare.myshopify.com is informed and verified once the feature ships in release 380/381 (Veena's commitment per comment #12)
- [ ] A "feature parity" audit of FedEx SOAP vs FedEx REST settings is captured somewhere durable, so the next SOAP→REST parity gap (e.g., ZI-508 Edit Package) doesn't surface customer-by-customer
- [ ] Customer-facing release note explicitly calls out buffer-days as restored on REST so other SOAP merchants don't hit the same gap silently

---

## Acceptance Criteria (Given/When/Then)

### Scenario: SOAP merchant who used buffer-days migrates to REST without manual reconfiguration

- **Given** a Shopify MCSL store with an active FedEx SOAP carrier configured with "Add Buffer Days to Estimated Delivery" set to e.g. 3 days
- **When** the merchant completes the FedEx REST migration (registers REST credentials and switches the rate-automation rule from SOAP to REST)
- **Then** the new FedEx REST carrier exposes the same buffer-days toggle in Carrier Settings → Other Details
- **And** the buffer-days value (3 days) is carried over from the SOAP setup so checkout continues to quote the padded estimate without a config gap

### Scenario: Net-new FedEx REST merchant can configure buffer days from day one

- **Given** a merchant with no existing FedEx SOAP carrier, registering FedEx REST for the first time
- **When** they open Carrier Settings → Other Details for the REST carrier
- **Then** the "Add Buffer Days to Estimated Delivery (upto 10 days)" toggle is present and persists across saves
- **And** the configured buffer is applied to the FedEx REST rate response's estimated delivery shown at Shopify checkout

### Scenario: Prevention — SOAP↔REST parity is auditable, not customer-discovered

- **Given** the next FedEx carrier feature is added to the SOAP integration UI
- **When** the engineering team ships that feature
- **Then** a parity check / shared config schema flags whether the REST integration exposes the same setting
- **And** if not, the gap is surfaced in product/eng review before a customer reports it via Zendesk
