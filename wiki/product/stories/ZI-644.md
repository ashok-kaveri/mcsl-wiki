## Problem Statement

FedEx REST carrier registration fails for any country where the postal code field is suppressed (`post: 'NONE'` in the countries referential).

```
instance.accountDetails.postalCode is not of a type(s) string
```

**Evidence**: [Ticket #396159](../../zendesk/summaries/396159.md) (ZI-644) — Kuwait merchant unable to register FedEx account after 10+ days and 3 support calls.

### Root Cause (two issues)

**Issue 1 — Type mismatch**: `Country.js:12` defines `const DEFAULT_POST_CODE = 99999` — a **number literal**. When a country has `post: 'NONE'`, `initializePostCode()` sets the form value to this number. It flows through `carrierRegistrationService.js:142` unchanged and gets rejected by the carrier-registration-api JSON schema (`postalCode: { type: 'string' }`). ~65 countries affected.

**Issue 2 — Kuwait incorrectly marked as no-postal-code**: `countries.js:141` has `{ label: 'Kuwait', value: 'KW', post: 'NONE' }`, but Kuwait has postal codes (e.g., `99923` per FedEx-registered address in Comment #37). The field is hidden entirely, so the merchant can't enter their actual zip code.

## Fix Design

| # | Fix | File | Change |
|---|-----|------|--------|
| 1 | Type coercion at API boundary | `server/src/modules/carrierRegistration/services/carrierRegistrationService.js:142` | `postalCode: String(zipCode)` |
| 2 | Kuwait postal codes | `client/src/components/common/country/referential/countries.js:141` | Remove `post: 'NONE'` from Kuwait entry |

| Repo | What changes |
|------|-------------|
| `storepep-react` | `carrierRegistrationService.js:142` — `String()` coercion; `countries.js:141` — Kuwait allows zip code entry |

## User Story

As a **merchant in Kuwait**,
I want **to enter my postal code during FedEx registration and have it accepted**,
so that **I can onboard FedEx and start generating shipping labels**.

### Acceptance Criteria

- [ ] Kuwait shows the postal code field in the registration form (no longer hidden)
- [ ] Merchant can enter their actual zip code (e.g., `99923`) for Kuwait
- [ ] Countries that genuinely have no postal codes → `postalCode` is coerced to string at server boundary, registration succeeds
- [ ] Countries with `postCode` override (HK, AE) → override string used (no regression)
- [ ] Countries with visible postal code field (US) → entered value passed as-is (no regression)
