## User Story

As a **paying WSS merchant in Estonia who ships internationally on FedEx every day and whose customs paperwork has to match the parcel's declared value to the cent**, I want **the app to send FedEx a customs value that is identical, to two decimal places, to the declared value it just calculated — instead of leaking a JavaScript float artifact like `127.39999999999999` into a regulated customs field and getting the label rejected by FedEx with a confusing "Total Declared value exceeds customs value" error**, so that **my team is not forced to abandon the plugin I'm paying $209/year for and walk the order over to FedEx's web portal one at a time on a normal Tuesday, my customers don't wait an extra day while we figure out why a sum that looks the same in two places is different in the API payload, and the same float-rounding bug doesn't quietly bite the next merchant whose cart totals happen to land on an awkward fraction.**

## Acceptance Criteria (Simple)

- [ ] Order #258403 on leyakoreya.ee (LEYAKO OU, account `512d0981-b53e-4eea-8c51-b9fed8788975`) generates a FedEx label successfully with customs value `127.40` and declared value `127.40` — no "Total Declared value exceeds customs value" error.
- [ ] Every numeric value the app sends to a FedEx label/customs payload — customs value, declared value, line-item value, insurance value, duty value — is rounded to exactly 2 decimal places at the serialization boundary, so JavaScript float artifacts (`.39999…`, `.20000…1`) can never reach the carrier.
- [ ] The same payload audit is applied to the other carriers wired into WSS (DHL on this account, and any others using customs/declared value) so we don't fix FedEx and leave the same bug live elsewhere.
- [ ] A regression test ships with the fix that constructs a cart whose line-item sum lands on a known float-imprecise total (e.g. `0.1 + 0.2`, or the actual `127.40` from this ticket), invokes the FedEx label flow, and asserts the outgoing payload contains the rounded value verbatim.
- [ ] When a future carrier rejection mentions "declared value" or "customs value" mismatch, the app's error surface tells the merchant which order field is being objected to and shows the value as the carrier saw it — so support does not need to pull logs to identify a rounding-class failure again.

## Acceptance Criteria (Given/When/Then)

### Scenario: LEYAKO OU regenerates Order #258403 after the fix lands

- **Given** Order #258403 on account `512d0981-b53e-4eea-8c51-b9fed8788975` has a line-item total that produces customs value `127.39999999999999` under current float arithmetic, and the declared value is calculated as `127.40`
- **When** the LEYAKO team retries label generation for that order with FedEx selected as the carrier
- **Then** the FedEx API payload contains `customsValue: 127.40` and `declaredValue: 127.40` — identical to two decimal places — and FedEx returns a label PDF
- **And** the customer can fulfil the order through WSS without exporting to FedEx's portal

### Scenario: A new merchant hits the same float pattern on a different cart

- **Given** a WSS merchant on FedEx has a cart whose subtotal happens to land on a float-imprecise value (e.g. three items at `42.47` each, or any quantity × unit-price that triggers IEEE-754 imprecision)
- **When** the merchant generates an international FedEx label
- **Then** the customs value, declared value, and any other money field in the payload are each rounded to 2 decimals before serialization, so the FedEx API never receives a 14-decimal float
- **And** no rounding-class rejection ("declared value exceeds customs value", "value mismatch", etc.) occurs for this merchant on this order

### Scenario: Prevent the next rounding bug from going to L3

- **Given** the customs/declared-value payload audit has shipped and the regression test covers float-imprecise totals
- **When** a future code change touches the WSS customs payload builder (FedEx, DHL, or any other carrier)
- **Then** the regression test fails the build if any money field can leak more than 2 decimal places into a carrier API request
- **And** if a carrier ever does reject a label with a value-related error, the app surfaces the exact field name and the exact value it sent — so the merchant or support can act without raising L3 against engineering
