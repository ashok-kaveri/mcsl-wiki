## Open Issues

1. **Adult Signature automation rule not working post-REST migration** — Automation rule signature option fails to apply ASR to labels after FedEx REST API migration. Labels generate with DSR (Direct Signature) instead of ASR (Adult Signature Required) when order total exceeds $100. Blocked: customer (daily operations impacted, cannot use automation rules for ASR). Severity: high (business-critical, wine retailer requires ASR for all shipments). Area: `carrier-migration`. (Comment #46, #79, #80)

2. **CSV signature update feature broken post-REST migration** — Product CSV import/update with signature option not applying Adult Signature to products. Customer attempted bulk CSV update but ASR not reflected on products. Blocked: customer (cannot use CSV for bulk ASR configuration). Severity: high (eliminates self-serve bulk update workaround). Area: `product-management`. (Comment #44, #46)

3. **Daily manual product ASR updates required** — Customer adds new wine products daily (new vintages replace old), each requiring manual ASR configuration by support team. Both automation rules AND CSV bulk updates are broken, leaving only manual one-by-one updates. Blocked: customer (ongoing manual dependency, no self-serve path). Severity: high (operational burden, scalability issue, blocks business growth). Area: `product-management`. (Comment #81, #82)

4. **Packaging type validation errors on label generation** — "Packaging type not allowed" error when generating labels with custom packaging and valid services. FedEx reports "One Rate" special service incorrectly passed in request. Blocked: engineering (code fix required). Severity: medium (intermittent label failures). Area: `label-generation`. (Comment #76, #79)

## User Story

As a **wine retailer who successfully completed the FedEx REST API migration but lost critical automation capabilities in the process**,
I want **my Adult Signature automation rules to apply ASR correctly to every wine shipment just like they did before the migration**,
so that **I don't have to manually request product updates from support every single day, my team can focus on growing the business instead of fighting broken automation, and I'm not legally exposed by shipping age-restricted products without proper signature requirements**.

## Acceptance Criteria (Simple)

- [ ] Automation rule ASR configuration applies correctly to labels generated after FedEx REST API migration
- [ ] Labels for wine products show "Adult Signature Required" not "Direct Signature Required" when ASR is configured via automation rule
- [ ] Pierre (grapebottle.myshopify.com) can generate labels for new wine products without daily manual product configuration by support
- [ ] Automation rule signature settings persist correctly through the REST API label request payload
- [ ] New merchants completing FedEx REST migration retain full automation rule functionality including signature options

## Acceptance Criteria (Given/When/Then)

### Scenario: ASR automation rule applies correctly post-migration (Happy path)
- **Given** a Shopify store has completed FedEx REST API migration
- **And** an automation rule is configured with "Adult Signature Required" for wine products
- **When** a wine product order is processed through automation rules
- **Then** the generated FedEx label shows "Adult Signature Required" (ASR) not "Direct Signature Required" (DSR)
- **And** no manual product configuration is needed

### Scenario: Automation rules work for all merchants post-migration (Broader fix)
- **Given** any merchant completes FedEx REST API migration with active automation rules
- **And** automation rules include signature requirements (ASR, DSR, or no signature)
- **When** labels are generated through automation rules
- **Then** the configured signature option applies correctly to every label
- **And** no support intervention is required to restore automation rule functionality

### Scenario: Migration regression testing prevents future breakages (Prevention)
- **Given** FedEx REST API integration code changes are deployed
- **When** QA regression suite runs
- **Then** automation rule signature options are explicitly tested on both SOAP and REST API paths
- **And** any deviation in signature behavior between SOAP and REST is flagged as a test failure before production
