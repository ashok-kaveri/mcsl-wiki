## Open Issues

1. **TNT Express label generation fails when product name contains '&' or exceeds API length limit** — TNT Express API rejects the `<Piece reference>` field when the value contains the `&` character or exceeds the maximum allowed length. The plugin populates this field from the order's product name and currently passes it through without sanitization. For single-product orders where the lone product's name contains `&`, every label fails. Multi-product orders happen to work because the plugin uses the first product's name (so the failure depends on product ordering). Custom Description override is documented but didn't resolve the case for this customer (root cause is not length, it's the special character). Temporary workaround: manually edit the product name on each order via the app's Edit Package UI — not viable at scale. Engineering scope: sanitize the piece-reference value (escape/strip `&` and other carrier-rejected characters; enforce per-carrier length limits). Blocked: engineering. Severity: high (every single-product TNT Express order with `&` in the product name fails; customer has multiple orders waiting and is asking for "urgency"; affects Portugal market where the merchant is active). Area: label-generation. (Comments #4, #8, #10, #11, #12)

## Resolved Issues

No resolved issues — the underlying sanitization gap is moving to `dev_needed`. Temporary manual per-order workaround is in place.

## User Story

As a **Portuguese publisher whose TNT Express labels fail on every single-product order where the product name contains the '&' character** — labels I cannot regenerate without manually renaming the product on every order — I want **WSS to sanitize the piece-reference value before sending it to TNT's API** (strip or escape `&`, enforce per-carrier length limits), so that **my labels print on the first try and I don't have to learn that '&' is the problem character through trial and error during a publishing rush**.

---

## Acceptance Criteria (Simple)

- [ ] TNT Express label requests sanitize the piece-reference value: special characters that TNT rejects (starting with '&') are escaped or stripped
- [ ] Length is enforced against TNT's per-field maximum so 'Piece reference has exceeded its maximum length' errors stop occurring
- [ ] Lighthouse Publishing's single-product '&' orders (e.g., #33972) succeed without manual product-name editing
- [ ] Sanitization is generalized — not TNT-specific — so other carriers with similar field constraints benefit
- [ ] The intermittent behavior (multi-product orders 'mysteriously' working because the first product's name was clean) no longer leaves customers confused about the root cause

## Acceptance Criteria (Given/When/Then)

- **Given** a UPS or FedEx order with '&' in product name

- **And** no over-sanitization regression on already-working carrier flows
