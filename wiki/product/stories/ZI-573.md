## User Story

As a **high-volume Indian confectionery merchant shipping 597 orders every two weeks through Amazon Express, whose 54 shipments got physically picked up but still show "Ready for Pickup" in MCSL for 6+ hours — making it impossible to confirm whether the courier actually collected the packages**,
I want **Amazon tracking status to sync into MCSL within 30 minutes of a carrier status change so I can see real-time shipment progress without checking Amazon's portal separately**,
so that **when shipments stall I know immediately whether it's a tracking delay or an actual pickup failure, and my team doesn't waste hours calling the Amazon FE to ask "did you actually pick these up?" when the answer is already in Amazon's system**.

## Acceptance Criteria (Given/When/Then)

### Scenario: Pickup status syncs within 30 minutes
- **Given** 54 Amazon Express shipments were physically picked up at 10:00 AM
- **When** Amazon's system updates their status to "Picked Up" at 10:05 AM
- **Then** MCSL reflects "Picked Up" for all 54 shipments by 10:35 AM
- **And** the order timeline shows the transition with Amazon's timestamp

### Scenario: Stale tracking retried automatically
- **Given** Amazon's tracking API returns "Ready for Pickup" for a shipment that was picked up 2 hours ago
- **When** MCSL's next sync cycle runs
- **Then** the system retries the API call with exponential backoff (5min, 15min, 30min)
- **And** once Amazon returns the updated status, MCSL updates immediately

### Scenario: Bulk tracking update
- **Given** a merchant has 80 Amazon shipments in various states
- **When** the tracking sync job runs
- **Then** all 80 shipments are checked and updated in a single batch operation
- **And** the sync completes within 5 minutes without API rate limit errors
