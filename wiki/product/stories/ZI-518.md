## User Story

As a **UK / European Shopify merchant who has a contracted account with APC for cost-effective domestic and Europe-bound parcel delivery**,
I want **APC available as a first-class carrier inside MCSL — rating at checkout, label generation from the order grid, manifest, and tracking — instead of being told my carrier isn't supported**,
so that **I can keep my Shopify fulfillment loop inside one app instead of bouncing to APC's portal for every label, my customers see APC rates at checkout, and I'm not forced to evaluate a competing shipping plugin just because PluginHive doesn't yet talk to a carrier I already pay for**.

---

## Acceptance Criteria (Simple)

- [ ] APC appears in the carrier-setup wizard with credential fields matching APC's REST API authentication (account number, API key/secret per APC's documentation)
- [ ] APC rates are returned for eligible Shopify orders and shown at checkout via the rate-fetch flow
- [ ] APC shipping labels generate from the order grid for both single-order and bulk flows, with the PDF / ZPL matching APC's accepted label format
- [ ] APC manifest closes successfully end-of-day, mirroring the manifest UX used by existing REST carriers (FedEx REST, DHL REST)
- [ ] APC tracking numbers post back to Shopify fulfillments and surface in the plugin's tracking page; test-credential round trip is documented in QA runbook for repeat validation

---

## Acceptance Criteria (Given/When/Then)

### Scenario: Merchant onboards APC for the first time
- **Given** a Shopify MCSL merchant has APC account credentials and selects APC in the carrier-setup wizard
- **When** they enter their APC REST API credentials and click "Validate"
- **Then** the plugin makes a sandbox auth call against APC's REST endpoint and confirms the account is active
- **And** the carrier is enabled for rate, label, manifest, and tracking flows without further configuration

### Scenario: Single-order and bulk label generation work for APC
- **Given** APC is enabled and a Shopify order is eligible for APC service
- **When** the merchant generates a label from the order grid (or in a bulk batch of 25+)
- **Then** the plugin calls APC's REST `ship` endpoint, receives a label PDF/ZPL, posts the tracking number to Shopify, and updates the order to `labelled`
- **And** failure modes (invalid postcode, unsupported service, account suspended) return human-readable errors rather than raw REST payloads

### Scenario: APC carriers stay parity-tested with the rest of MCSL going forward
- **Given** APC is a live carrier in production
- **When** the broader plugin gets new shipment features (return labels, third-party billing, multi-package shipments) or REST contract changes are announced by APC
- **Then** APC is included in the carrier-regression matrix alongside FedEx REST, DHL REST, and other live integrations
- **And** APC's credential-test runner runs in CI with the 1Password-provisioned sandbox account, so a credential rotation or contract change is caught before customers experience it
