## User Story

As a **non-profit merchant on the $209 Yearly Starter plan using WSS for daily shipping**,
I want **the Import Orders button to never tell me I've hit a plan limit when my plan and my usage don't actually breach any limit**,
so that **my manual fallback (when auto-import is already broken) isn't blocked by a bogus quota error, and I'm not forced onto a higher plan just to work around a bug**.

---

## Acceptance Criteria (Simple)

- [ ] When Gabriel clicks Import Orders on scoutsducanada.ca, the "As per your Subscription plan, orders import maximum limit has been reached" error does not appear unless he has genuinely exceeded the Starter plan's documented import limit
- [ ] The actual Starter-plan import limit is documented and matched against the customer's current usage to confirm whether a limit exists at all
- [ ] If a real limit exists and is hit, the error message shows current usage, the limit, when it resets, and a link to upgrade — not a vague generic error
- [ ] A telemetry signal is recorded every time this error fires, with plan, store, and usage snapshot, so product can see if other merchants are hitting it spuriously
- [ ] Gabriel is followed up with the root cause and apology for the 6-day SLA breach on a paying $209/year customer

---

## Acceptance Criteria (Given/When/Then)

### Scenario 1: No false limit error for Scouts du Canada
- **Given** Gabriel is on the Starter plan and his current month's order-import count is below the plan limit
- **When** he clicks Import Orders
- **Then** orders import without a "subscription plan limit reached" error
- **And** the import completes successfully

### Scenario 2: Genuine limits are explained clearly
- **Given** any merchant's plan genuinely enforces an import limit and they exceed it
- **When** the import is blocked
- **Then** the error states current usage, the configured limit, reset time, and an upgrade path
- **And** the merchant can click through to upgrade without opening a support ticket

### Scenario 3: Plan-gated errors are monitored and don't silently spike
- **Given** the plan-limit error path is instrumented
- **When** any merchant hits it
- **Then** an event is emitted with plan, store, and usage
- **And** product has a dashboard alert if false-positive rates climb, so we catch "error is firing despite usage being below limit" regressions before customers do
