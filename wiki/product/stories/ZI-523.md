## User Story

As a **fast-growing Indian D2C merchant on the $19 starter plan who just learned the hard way that Amazon Shipping doesn't cover every pincode**,
I want **the error MCSL shows when an Amazon label fails to tell me exactly which constraint Amazon rejected — pincode out of coverage, weight too high, dimensions too large — instead of a generic "No Shipping Service selected" that sends me hunting through WhatsApp messages to figure out what to do**,
so that **my packer at the warehouse can resolve a failure in seconds without opening a support ticket, and I can decide on the spot whether to retry with a different carrier, change the package size, or apologise to the customer — without losing trust in the app on day 10**.

---

## Acceptance Criteria (Simple)

- [ ] When Amazon Shipping India returns an "out of coverage / unsupported dims / unsupported weight" rejection on label generation, MCSL surfaces a clearly named error that names the rejection reason and the violating field (pincode, weight, or dim), not the generic "No Shipping Service selected" message Anandhaas currently sees
- [ ] The error message text used today for "No Shipping Service selected" is reserved only for the case where the merchant truly hasn't picked a service in their config — never used as a fallback for upstream carrier rejections
- [ ] Anandhaas's existing failing orders, retried after the fix lands, show the correct, actionable error string (verified by reproducing one of their out-of-coverage pincodes)
- [ ] The same error-mapping logic applies to every Amazon Shipping India merchant — there is no per-store config that toggles which error string the merchant sees
- [ ] Each mapped error includes a one-line suggested action ("try a different carrier", "reduce package weight", "split into multiple boxes") so the packer can self-resolve

---

## Acceptance Criteria (Given/When/Then)

### Scenario: Anandhaas hits an out-of-coverage pincode and gets an actionable message
- **Given** Anandhaas is generating a label on Amazon Shipping India for an order whose destination pincode is outside Amazon's serviceable footprint
- **When** the packer clicks "Generate Label" in the MCSL UI
- **Then** the error banner reads something like "Amazon Shipping cannot deliver to PIN <pincode> — try a different carrier" — naming the rejection reason and the violating field — and does NOT show the generic "No Shipping Service selected" string

### Scenario: Weight/dimension rejections are distinguished from coverage rejections
- **Given** an Amazon Shipping India order whose package weight exceeds Amazon's per-parcel maximum (or dimensions exceed limits)
- **When** the packer attempts label generation
- **Then** the error names "weight" (or "dimensions") as the violating field, suggests "reduce package weight" (or "split into multiple boxes"), and is visually distinct from the coverage-rejection message

### Scenario: Genuine config errors still surface as config errors
- **Given** a merchant who has set up the Amazon Shipping carrier but has not selected any service from the service-list under that carrier
- **When** they attempt label generation
- **Then** the error correctly reads "No Shipping Service selected" — because that's the actual problem — and links to the carrier config page; this case is the ONLY remaining caller of that error string
