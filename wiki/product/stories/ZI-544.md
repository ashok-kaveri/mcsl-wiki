### Open Issues (all four covered by this card)

1. **Large batch creation: missing redirect to Label Batch page; misleading "Error in Batch Creation" message** — For 1000+ order batches, batch creation takes a long time and the app fails to auto-redirect to the Label Batch page (works correctly for small batches). During the delay, the app shows "Error in Batch Creation," which is misleading since the batch is often still processing successfully — confusing customers. Behavior captured in a video shared earlier. Blocked: engineering (dev_needed; q4_dec_end). Severity: P2 — directly affects high-volume customers' confidence in batch operation. Area: `label-generation`. (Comment #1)

2. **Failed labels appear as "Processing" in original batch AND in retry batch; retried carrier overwrites original batch** — Workflow: 2 of 100 labels fail; customer retries those 2 with a different carrier. Bug: the failed orders show "Processing" in BOTH the original batch and the new retry batch simultaneously; additionally, the retried carrier's name gets written back into the original batch, corrupting batch metadata. Dev team has previously noted possible printing inconsistencies tied to the same state-tracking bug. Blocked: engineering (dev_needed). Severity: P2 — data-correctness issue across retry workflow; affects audit/print paths. Area: `label-generation`. (Comment #1)

3. **Print mismatch: "Not in Ship" order had a generated label and was fulfilled in Shopify** — Order #45279880 was excluded from Label Print because its status was "Not in Ship" (per Deepak), yet a label was actually generated and Shopify shows the order as fulfilled. The Amazon tracking ID (https://track.amazon.in/tracking/369348516162) is valid. First observation of this state inconsistency; root cause unknown. Blocked: engineering (root-cause investigation). Severity: P2 — data integrity / fulfillment status drift; risks repeating across other orders silently. Area: `order-management`. (Comment #1)

4. **Cascading priority-based carrier routing across Amazon / Delhivery / XpressBees** — Customer's working pattern is: try Amazon first → switch failed orders to Delhivery → retry failed Amazon with Amazon again → fall back to XpressBees on remaining failures. They want this multi-step priority cascade automated rather than manual. Blocked: product/engineering (dev_needed; q4_dec_end). Severity: P2 feature-request — affects daily operations of an active high-volume merchant. Area: `rate-shopping`. (Comment #1)

## User Story

As a **high-volume Indian D2C merchant running 1000+ order batches every morning across Amazon Shipping, Delhivery, and XpressBees — exactly Love of India's pattern — where the difference between "ship today" and "ship tomorrow" is whether MCSL's batch UI lies to me about what happened**,
I want **MCSL to (a) actually tell me my 1000-order batch is processing and redirect me to the batch page like it does for small batches, (b) keep retry batches and original batches as cleanly separate ledgers so a 2-order retry on Delhivery doesn't overwrite which carrier my original 100-order Amazon batch used, (c) never let an order show "Not in Ship" while a label is generated and Shopify is already fulfilled — that data divergence is silent and dangerous, and (d) let me declare my Amazon→Delhivery→Amazon-retry→XpressBees cascade as an automation rule instead of doing it by hand at 6am**,
so that **my packing team trusts the batch UI again, my finance team's reconciliation between Shopify fulfilment and MCSL labels actually matches at month-end, and my morning queue runs end-to-end in 20 minutes instead of being a manual carrier-juggling exercise every single day**.

## Acceptance Criteria (Simple)

- [ ] When a merchant submits a 1000+ order batch, MCSL auto-redirects to the Label Batch page within 3 seconds (matching the small-batch behaviour) and the page shows live progress — no misleading "Error in Batch Creation" appears while the batch is in fact processing
- [ ] When a merchant retries failed labels from an original batch by selecting a different carrier, the original batch's carrier name is NOT overwritten — the retry creates a clean separate batch ledger, and the failed orders appear with status "Failed → retried in batch #N" in the original (not still "Processing")
- [ ] No order can simultaneously be in MCSL state "Not in Ship" AND have a generated label AND be marked fulfilled in Shopify — at least one of those three has to flip, with a server-side invariant check that surfaces any drift case to support before customers see it
- [ ] A new automation-rule type "Carrier cascade" lets the merchant declare an ordered list of carriers (Amazon → Delhivery → Amazon retry → XpressBees), with per-step retry counts; the rule applies on bulk label generation without manual carrier switching
- [ ] All four behaviours are verified against a reproduction of Love of India's actual scenario (1000+ order batch, mid-batch retries with a different carrier, the exact #45279880 "Not in Ship" state) before the customer is contacted with a fix announcement

## Acceptance Criteria (Given/When/Then)

### Scenario: 1000-order batch shows accurate status, not a fake error
- **Given** a Shopify merchant with 1100 orders selects them all and clicks "Create Batch" in MCSL
- **When** batch creation begins and takes longer than the small-batch threshold to complete
- **Then** MCSL redirects the merchant to the Label Batch page (same redirect path as small batches), the page shows a clear "Processing X of 1100 — labels printing live" indicator, and the merchant never sees an "Error in Batch Creation" message during a batch that is in fact processing normally

### Scenario: Retry on a different carrier doesn't corrupt the original batch
- **Given** an MCSL merchant ran a 100-order Amazon Shipping batch, 2 labels failed, and they trigger a retry for those 2 orders on Delhivery
- **When** the retry batch completes successfully
- **Then** the original batch's carrier metadata still reads "Amazon Shipping" (not "Delhivery"), the 2 retried orders appear in the original batch with status "Failed → retried in batch #<retry_id>" linking to the retry batch, and the retry batch is a clean standalone record showing only those 2 orders shipped via Delhivery

### Scenario: Cascade routing automates Love of India's morning playbook
- **Given** a merchant has configured an automation rule "Cascade: Amazon → Delhivery → Amazon retry → XpressBees", and runs a 500-order bulk label generation
- **When** the bulk run completes
- **Then** every order has shipped on the first carrier in the cascade that returned a successful label, MCSL produced a single summary report ("420 via Amazon, 45 via Delhivery, 30 via Amazon retry, 5 via XpressBees, 0 failed"), and no order is left in "Processing" state in the original batch waiting for a manual retry

### Scenario: "Not in Ship" + label + fulfilled state divergence cannot happen silently
- **Given** any order in any MCSL store that ends up in a state where it has a generated label, is marked "Not in Ship", and is fulfilled in Shopify
- **When** the state writes happen (in any order — label gen, status change, Shopify webhook)
- **Then** MCSL's invariant check catches the divergence at write time, flips the "Not in Ship" status to "Shipped" (or surfaces the conflict to support if auto-recovery isn't safe), and order #45279880 — the original observed case — does not have any same-pattern peers in the same store after the fix lands
