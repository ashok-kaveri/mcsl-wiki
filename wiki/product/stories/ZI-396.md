## User Story

As a **French artisan brand merchant running 200+ international orders/month through UPS** out of France, I want **UPS WorldEase consolidated-customs API integration in the app — the same WorldEase service my UPS account contract already includes**, so that **my multi-package international consignments clear customs once instead of per-parcel, my customs fees drop accordingly, and I don't have to drop out of WSS into a separate UPS portal for every WorldEase shipment**.

## Acceptance Criteria (Given/When/Then)

### Scenario: Merchant ships a consolidated international movement with WorldEase

- **Given** a French merchant with a UPS account that has WorldEase entitlement and 5 international orders ready for label generation
- **When** they select WorldEase as the service and generate labels for all 5 orders in one batch
- **Then** the app creates one master/docbox shipment, attaches all 5 child labels via the same GCCN, and customs is filed once for the master
- **And** within 7 days the app automatically closes the WorldEase movement (or surfaces a clear "close pending" warning)

### Scenario: UPS WorldEase entitlement is missing from the customer account

- **Given** a merchant attempts to use WorldEase but their UPS account does not have the entitlement
- **When** they try to generate a WorldEase label
- **Then** the app surfaces a specific, actionable error pointing the customer to contact their UPS rep for the agreement, rather than a generic API failure
- **And** the error message names 'WorldEase' explicitly so the customer doesn't open a support ticket asking what went wrong

### Scenario: Token-lifetime change on 2026-01-04 does not break WorldEase calls

- **Given** UPS has reduced OAuth token validity from 4h to 1h as of 2026-01-04
- **When** the app makes consecutive WorldEase API calls across a 90-minute label-generation batch
- **Then** the app refreshes the OAuth token transparently and no labels fail with auth-expired errors
- **And** a token-refresh observability signal is logged so support can debug any future auth regression
