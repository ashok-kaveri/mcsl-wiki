## User Story

As a **WooCommerce merchant on a $9 plan running a Bahrain-based store where DHL is only viable for international and I need an affordable domestic carrier for local deliveries**,
I want **TryParcel available as a supported carrier in PluginHive WSS so I can generate labels and show live rates for Bahrain-domestic orders directly from my store**,
so that **I stop managing domestic shipments outside the app with manual label entry, and my Bahrain customers see correct domestic rates at checkout instead of inflated DHL international rates**.

---

## Acceptance Criteria (Simple)

- [ ] Siam Computing's store can connect a TryParcel account (API key / credentials) from the carrier-config page
- [ ] Bahrain-domestic rates from TryParcel surface at checkout alongside DHL rates, correctly priced
- [ ] Labels can be generated for TryParcel shipments with correct format (domestic Bahrain format) and shipment data is pushed to TryParcel's API
- [ ] PluginHive has documented internal feasibility decision on TryParcel — either committed integration (with ETA) or declined with rationale — communicated to customer within 14 days of receiving carrier contact
- [ ] A standardized "request new carrier" flow captures carrier URL, merchant volume, API-docs availability, and carrier-contact info so future regional-carrier requests don't require repeated ad-hoc collection

---

## Acceptance Criteria (Given/When/Then)

### Scenario: Deliver TryParcel integration for Siam Computing
- **Given** Siam Computing has provided TryParcel API documentation and a technical contact, and the integration has been scoped and built
- **When** the merchant adds TryParcel credentials on the Add Carrier page and places a Bahrain-domestic order
- **Then** TryParcel domestic rates appear at checkout
- **And** the label for that order is generated via the TryParcel API and appears in the shipment's Shipping section

### Scenario: Scale the same pattern to future regional carriers
- **Given** any merchant requests a new regional carrier via the in-app Add Carrier form
- **When** the request is submitted
- **Then** the request is captured with all required fields (URL, volume, API docs, carrier contacts)
- **And** product team has a dashboard view of open new-carrier requests sorted by total customer demand / plan tier

### Scenario: Keep requesting customer informed without manual follow-up
- **Given** a new-carrier request is under evaluation
- **When** the status changes (scoped / committed / declined)
- **Then** the requesting customer is automatically notified by email and sees the status in-app
- **And** if declined, the customer is offered either a generic API-connector fallback or a paid custom-integration quote (following the Delivro pattern)
