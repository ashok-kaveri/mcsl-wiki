## Open Issues

1. **WooCommerce Composite Products not supported by product sync or packaging** — Plugin only supports Simple and Variable WooCommerce product types. Customer's two Composite Products ("You Choose" 3-Soup and 6-Soup Gift Boxes) don't sync; workaround is to temporarily convert Composite → Simple → Composite to force initial sync. Even after the sync hack, the plugin packs the gift-box components as separate items rather than recognizing the composite's box, which can produce wrong rates when multiple composites ship together (especially during gift-heavy holidays). Same Composite Products requirement as ZI tickets #350796 (Greaves Jams). Blocked: engineering. Severity: high (holiday-season risk, multiple customers affected, on the MCSL 378 plan). Area: product-management. (Comments #18, #20, #25, #27, #29, #32)

2. **Role-restricted products skipped during bulk WooCommerce import** — When a store uses "Product Visibility by User Role for WooCommerce" (or similar visibility plugin) to restrict products to specific user roles (Wholesale here), the WooCommerce REST API does not return those products to WSS during bulk product fetch. WSS shows partial catalog (40/56 here) without explaining why. Per-product webhook updates DO sync the missing products, but bulk import doesn't. Customer was forced to disable the visibility plugin in maintenance mode to import. Blocked: engineering. Severity: medium (silent partial sync that takes hours of diagnosis to identify; affects any store using role-based product visibility plugins). Area: product-management. (Comments #9, #15, #16, #18)

3. **APO addresses rejected via Stamps USPS API** — Customer's APO orders (US military overseas addresses, treated as domestic by USPS) get an `Address Not Valid` error in WSS, even though USPS.com accepts the same address for Ground Advantage and Priority Mail. Support workaround: generate APO labels directly on Stamps.com. Customer feedback: this is a documentation/UX gap — the plugin shouldn't return a confusing "Address Not Valid" when the underlying issue is API capability. Blocked: engineering / Stamps API. Severity: low (workaround exists; APO is a narrow use case). Area: label-generation. (Comments #19, #20)

4. **Products page "Import" button labeling misleading** — UI label says "Import Products" on the Products page CSV controls, but the function is "Update already-synced products" — caused customer to attempt manual product import and fail with confusing errors. Blocked: engineering / UX. Severity: low (one-off customer confusion; minor UI rename). Area: product-management. (Comments #9, #10)

## User Story

As a **WooCommerce agency owner managing multiple food-retail stores with wholesale, retail, and gift-box product lines across different visibility roles**,
I want **my entire product catalog — including Composite Products and role-restricted items — to sync into the shipping plugin automatically during bulk import, with composite packaging recognized as a single box rather than individual components**,
so that **I stop burning hours on workaround hacks (toggling product types, disabling plugins in maintenance mode) every time I onboard a new client store, and my client's holiday gift-box orders ship at the correct rate instead of being overcharged by component-level packing logic**.

---

## Acceptance Criteria (Simple)

- [ ] WooCommerce Composite Products are fetched and imported during bulk product sync alongside Simple and Variable product types
- [ ] Composite Product components are packed as a single composite unit (using the parent product's dimensions and weight) rather than as separate individual items during rate calculation
- [ ] Products restricted to specific WooCommerce user roles via visibility plugins are included in the bulk product import API response, or the system falls back to an admin-context API call that bypasses role restrictions
- [ ] When bulk product sync results in a partial catalog (fewer products imported than published), the Products page displays a warning banner explaining how many products were skipped and why
- [ ] The Products page CSV "Import" button is relabeled to "Update Products" (or equivalent) to distinguish it from the initial bulk sync function
- [ ] Stores with existing Composite Products imported via the Simple-then-convert workaround retain their catalog entries after the fix is deployed, without requiring re-import

## Acceptance Criteria (Given/When/Then)

### Scenario: Composite Products sync during bulk import
- **Given** a WooCommerce store has published Composite Products (e.g., "You Choose 3-Soup Gift Box" and "You Choose 6-Soup Gift Box") alongside Simple and Variable products
- **When** the store owner triggers a bulk product sync from the WSS Products page
- **Then** all product types including Composite Products appear in the synced catalog with their parent-level dimensions, weight, and component list intact

### Scenario: Composite packaging uses parent box dimensions for rate calculation
- **Given** a customer orders two Composite Products (each a gift box containing 3 or 6 soup items) in a single order
- **When** the system calculates shipping rates for that order
- **Then** each Composite Product is packed as one unit using the parent product's declared box dimensions and weight, not as 3 or 6 separate soup items, producing an accurate rate quote

### Scenario: Role-restricted products are included in bulk sync via admin-context fallback
- **Given** a WooCommerce store uses "Product Visibility by User Role" to restrict 16 Wholesale-category products to the `wholesale` role only, and the store has 56 total published products
- **When** the store owner runs a bulk product sync from WSS
- **Then** all 56 products are imported (including the 16 role-restricted ones), and the sync summary shows 56/56 products imported with no silent omissions
