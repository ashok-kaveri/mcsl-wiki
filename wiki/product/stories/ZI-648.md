## User Story

As a **WooCommerce merchant who set up a checkout shipping option called "Deliveroo" (a colloquial name for Delivro used in Quebec)**,
I want **the plugin to recognize "Deliveroo" as a valid alias for the Delivro carrier when mapping checkout shipping methods to carriers for label generation**,
so that **orders placed under my "Deliveroo" shipping option are automatically routed to Delivro without requiring me to rename my checkout methods or create manual workaround automation rules for every service variation**.

---

## Acceptance Criteria (Simple)

- [ ] The plugin recognizes "Deliveroo" (and common variations like "deliveroo", "DELIVEROO") as an alias for the Delivro carrier during checkout-to-carrier mapping
- [ ] A configurable carrier alias mapping exists so merchants can define their own custom shipping method names that resolve to specific carriers without code changes
- [ ] When a WooCommerce flat rate method named "Deliveroo" is selected at checkout, the resulting order is pre-mapped to Delivro in the plugin's order view
- [ ] The alias mapping does not interfere with existing automation rules — if both an alias match and an automation rule could apply, the automation rule takes precedence
- [ ] The carrier alias configuration is accessible from the carrier settings page within the plugin, not buried in developer-only configuration files

---

## Acceptance Criteria (Given/When/Then)

### Scenario: Checkout method with carrier alias maps correctly
- **Given** a WooCommerce store with Delivro configured and a flat rate shipping method named "Deliveroo" ($12.99) visible to customers at checkout
- **When** a customer completes checkout selecting the "Deliveroo" shipping option and the order appears in the WSS plugin
- **Then** the plugin automatically maps the order to the Delivro carrier based on the "Deliveroo" alias, and the merchant sees "Delivro" pre-selected as the carrier in the order's shipping details

### Scenario: Merchant configures custom alias for a carrier
- **Given** a merchant navigates to the Delivro carrier settings page in the WSS plugin and sees a "Shipping Method Aliases" configuration field
- **When** the merchant adds "Deliveroo" and "Livraison Delivro" as custom aliases for the Delivro carrier and saves the configuration
- **Then** any future orders using WooCommerce shipping methods with those exact names (case-insensitive) are automatically mapped to Delivro without requiring separate automation rules

### Scenario: Automation rules take precedence over alias matching
- **Given** the merchant has both a carrier alias ("Deliveroo" mapped to Delivro) and an automation rule that maps "Deliveroo" orders over 10kg to CanPar instead
- **When** a customer places an order weighing 15kg and selects the "Deliveroo" shipping option at checkout
- **Then** the automation rule fires first and assigns the order to CanPar, overriding the alias match, and the plugin logs which rule was applied for support traceability
