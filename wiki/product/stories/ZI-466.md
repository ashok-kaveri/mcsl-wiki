## User Story

As a **support engineer trying to resolve a time-sensitive carrier integration issue for a merchant with daily dispatch deadlines**,
I want **direct escalation paths to carrier technical teams that don't require the merchant to be the middleman**,
so that **I can diagnose and resolve label failures within the 72-hour SLA instead of waiting days for carrier responses while the merchant's operations are blocked**.

## Acceptance Criteria (Given/When/Then)

### Scenario: Direct escalation to CouriersPlease for API issues
- **Given** support identifies a CouriersPlease API issue (403, 500, timeout, ambiguous error)
- **When** L3 creates an escalation
- **Then** escalation goes directly to CouriersPlease technical/API team (not general support)
- **And** escalation includes: merchant account ID, error logs, API request/response payload, expected vs actual behavior

### Scenario: Self-service diagnostics for common carrier errors
- **Given** merchant experiences a label failure with error "403 Forbidden" from any carrier
- **When** merchant views the error in the app
- **Then** app displays self-service diagnostic steps: (1) Test API credentials, (2) Verify account status with carrier, (3) Check service availability for destination postal code, (4) View API rate limits/throttling status
- **And** merchant can retry with diagnostic mode enabled (detailed logging for support)

### Scenario: Carrier escalation SLA tracking
- **Given** support escalates an API issue to any carrier (CouriersPlease, USPS, FedEx, etc.)
- **When** escalation is created
- **Then** system tracks: escalation timestamp, expected carrier response time (per carrier SLA), actual response time, resolution timestamp
- **And** if carrier SLA is breached, support is alerted to follow up or provide merchant workaround
