### Phase 1: Initial FedEx REST Migration Issue (2026-03-17)
- Customer migrated to FedEx REST API as requested by PluginHive
- FedEx rates stopped appearing at checkout after migration
- Support confirmed rates were working, suggested checking automation rules (Comment #2-3)

### Phase 2: Intermittent Rate Failures (2026-03-19 to 2026-03-20)
- Customer reported rates work "sometimes but not all the times"
- Fallback to manual backup prices instead of dynamic FedEx rates
- Customer hypothesis: Too many UPS+FedEx rates causing Shopify timeout — reduced rate count, issue persisted (Comment #6-8)
- Support removed and re-added carrier services, but 2 out of 3 tests still timed out (Comment #12-14)

### Phase 3: L3 Escalation (2026-03-20 to 2026-03-31)
- Escalated to L3: Rates returning successfully in logs but not displaying at checkout (Comment #16)
- Customer reported 2 orders over weekend both used backup rates (Comment #17)
- One week wait with persistent failures — customer escalated urgency (Comment #19, #21)
- **L3 identified root cause**: Token refresh timing issue (Comment #37)
- Fix deployed to QA on 2026-03-31, released 2026-04-01 (Comment #25-28)

### Phase 4: Fix Verification & Ongoing Monitoring (2026-04-02 to 2026-04-16)
- Customer reported fix did NOT resolve issue: 1 out of 5 tests showed correct rates (Comment #29)
- Support verified rates working on multiple test checkouts (Comment #30-31)
- Customer provided evidence: 5 orders since Friday, 4 used backup rates (Comment #32)
- **L3 re-investigation**: Customer had disabled automation rule; L3 re-enabled and monitored (Comment #36)
- **L3 final diagnosis**: "Very specific case where it happens during a token refresh... Will need some dev to fix this" (Comment #37)
- Support confirmed no failures during 3-day monitoring period (Comment #38)

Severity: High (revenue impact from incorrect shipping charges, customer frustration after 30-day unresolved case). Area: `carrier-migration`. (Comment #37)

Area: `carrier-migration`. (Comment #36)

## User Story

As a **store owner who uses automation rules to filter which carrier rates display at checkout and was recently migrated to FedEx REST API**,
I want **my automation rule configuration to persist reliably after FedEx REST migration without unexpectedly disabling itself**,
so that **I don't have to contact support to manually re-enable rules I already configured, my checkout continues showing only the rates I want customers to see, and I can trust that REST API migration won't silently break settings I depend on**.

---

## Acceptance Criteria (Simple)

- [ ] Automation rules remain enabled and active after FedEx REST API migration
- [ ] System does not silently disable automation rules during or after migration
- [ ] If automation rules are disabled, clear error message explains why and how to fix
- [ ] Green Hockey AG store (#379340) maintains automation rule configuration without L3 intervention
- [ ] Post-migration audit identifies any stores with disabled automation rules and re-enables them

---

## Acceptance Criteria (Given/When/Then)

### Scenario: Automation rules persist through FedEx REST migration
- **Given** Green Hockey AG has automation rules configured before FedEx REST migration
- **When** the store is migrated from legacy FedEx API to REST API
- **Then** automation rules remain enabled and active
- **And** checkout rates continue filtering according to the configured rules

### Scenario: Clear error messaging if rules disabled
- **Given** automation rules are disabled for any reason (migration conflict, API incompatibility, etc.)
- **When** store owner views the automation rules settings page
- **Then** a clear error message explains why rules are disabled
- **And** the message provides actionable steps to re-enable them

### Scenario: Prevent silent configuration loss
- **Given** store owner configured automation rules weeks or months ago
- **When** FedEx REST migration occurs
- **Then** no automation rule settings are lost or silently disabled
- **And** store owner does not need to re-configure or manually re-enable anything
