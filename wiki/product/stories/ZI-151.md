# Ticket #361776 — Package Split Logic & Declared/Insured Value After Manual Edit

- **Customer**: Valeria Kolobova, Sales Support Team Lead (Emlid)
- **Duration**: 2025-11-27 to 2026-02-20 (ongoing)

## Timeline & Key Phases

### Phase 1: Package Split & Unit System Issues (2025-11-27)
- Customer placed test order #1413 with 6x SPL102 and 16x RCH205-I
- Expected 3 boxes but app produced 2 boxes
- Box selection incorrect: RCH205-I allocated to "Generic (config 2)" instead of the dedicated RS3 box
- Customer noted the demo store uses pounds/inches while their store uses kg/cm, requesting testing in metric units (Comment #1)

### Phase 2: Declared Value / Insurance Bug (2025-11-27)
- L3 identified a separate issue: discounted product price (set at 35% declared value under App Settings > General Settings > Shipping) is not applied when packages are manually edited (Comment #5)
- Example: product priced $100 with 50% checkout discount should declare $50, but after manual package edit the original $100 value is used for insurance calculation
- Video recording shared documenting the bug (Comment #6)

### Phase 3: Enhancement Committed (2025-11-28)
- Senior team confirmed enhancement to maintain 35% declared value calculation after manual package edits (Comment #7)
- Tentative ETA: end of December 2025 or mid-January 2026 (Comment #12)
- Customer asked for clarity on whether both package split and declared value would be addressed (Comment #9)
- Support confirmed only the declared value enhancement is scoped (Comment #10)

### Phase 4: Package Split Investigation (2025-12-08 to 2026-01-07)
- Customer followed up on package split issue (Comment #14)
- Support explained the app's bin-packing algorithm: it optimizes for fewest boxes, not product-to-box mapping (Comment #20)
- Customer explained their real-world constraint: products come in manufacturer boxes (6-unit and 8-unit) and must ship in those specific boxes (Comment #21)
- Support confirmed product-specific box assignment rules are not supported; packaging settings are global (Comment #22)
- Customer asked about workarounds using product dimension manipulation; support tested and confirmed this does not work (Comment #27)
- Enhancement for declared value confirmed for release 371a, ETA 2-3 weeks from 2026-01-07 (Comment #26)

### Phase 5: Delays & Follow-ups (2026-01-09 to 2026-02-20)
- Customer asked for updates; release delayed (Comment #29)
- Support confirmed fix is in final testing stage (Comment #30)
- Release delayed again; new tentative ETA around mid-March 2026 (Comment #34, #36)
- Customer acknowledged and expressed gratitude (Comment #37)

## Open Issues

1. **Declared/insured value resets after manual package edit** — When a package is manually edited after order creation, the 35% declared/insured value calculation based on discounted price is lost and reverts to the full product price. Blocked: dev team (in final testing, tentative ETA mid-March 2026). Severity: high, causes incorrect insurance values on international shipments. Area: international. (Comment #36)
2. **No product-to-box mapping rules** — The app's packaging algorithm globally optimizes for fewest boxes; it cannot assign specific products to specific manufacturer boxes. This is a design limitation, not a planned enhancement. Blocked: architecture constraint. Severity: moderate, requires manual package editing as workaround. Area: label-generation. (Comment #22)

## Resolved Issues

None fully resolved; declared value fix is in final testing.

## Customer Context

- Emlid is a technology/surveying equipment company (international shipper)
- Uses UPS carrier for international shipments
- Products have specific manufacturer packaging (6-unit and 8-unit boxes)
- Relies on 35% declared value for insurance calculations on discounted orders
- Very engaged, detail-oriented customer who tracks ETAs closely
- Sales Support Team Lead handling logistics; professional communication style
