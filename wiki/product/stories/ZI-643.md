## Problem Statement

India Post booking requires a `barcode_no` field — the tracking/article number. DoP does **not** auto-allocate; the caller must supply a unique, valid barcode. The current implementation generates it via `Math.random()` in `generateBarcode()` (`requestBuilder.js:34`), which has three critical defects:

1. **Collision risk** — `Math.random()` is not unique. Duplicate barcodes trigger DoP rejection: `"Duplicate article: Already booked today or yesterday"`. Risk compounds across PM2 workers sharing no state.
2. **Invalid S10 format** — Production enforces UPU S10 standard (13-char: 2-letter prefix + 8-digit serial + 1 check digit + 2-letter country code `IN`). `Math.random()` produces neither valid check digits nor valid prefixes.
3. **Sandbox masks the bug** — `test.cept.gov.in` accepts invalid check digits and junk prefixes, so the bug is latent in dev/QA but will fail in production.

### Verified against India Post sandbox

- `barcode_no` is **mandatory** — omitting it errors `"Barcode number is required"`
- **Uniqueness hard-enforced**: duplicate → `"Duplicate article: Already booked today or yesterday"`
- `block_number`/`offset_number` in the response are DoP-side batch counters, not a series we control
- Sandbox creds/contracts in `~/pluginhive-skill-gen/attempted-refactor/assets/india-post/credentials.md` (customer/bulk `3000064781`, contract `41585456` SP_INLAND)

## Root Cause

`server/src/shared/API/carriers/indiaPost/requestBuilder.js:34` — `generateBarcode()` uses `Math.random()` to produce the barcode. No persistence, no uniqueness guarantee, no S10 compliance.

## Fix Design

Replace `Math.random()` with a **persisted monotonic serial counter** on the India Post carrier record:

1. **Atomic counter** — `$inc` on the carrier document (MongoDB) to get the next serial. Must be atomic (unlike Xpressbees' racy read-modify-write pattern in `awbInfo`/`getEnhancedCarriersForXpressBees`).
2. **S10 check-digit helper** — compute check digit per UPU S10 standard: weights `[8,6,4,2,3,5,9,7]`, mod 11, `11−r`, `10→0`, `11→5`.
3. **Carrier field** — add `indiaPostBarcode` (or similar) counter field to the carrier record schema. Currently no India Post-specific carrier fields exist; `awbInfo` is Xpressbees-only.
4. **Prefix/range** — open question for CEPT (Ratheesh, `integrations.cept@indiapost.gov.in`): must the barcode come from a DoP-allotted block/prefix range tied to the contract, or does prod accept any unique valid S10 under a service prefix? No allocation endpoint among the 10 APIs → likely handed over offline.

## User Story

As a **merchant using India Post through MCSL**,
I want **every booking to generate a unique, DoP-compliant tracking number automatically**,
so that **my shipments don't fail with duplicate-article errors and my tracking numbers are valid in production**.

### Acceptance Criteria

- [ ] Given a booking request, when `barcode_no` is generated, then it is a valid S10 barcode (2-letter prefix + 8-digit serial + valid check digit + `IN`)
- [ ] Given concurrent bookings across PM2 workers, when each calls generateBarcode(), then no two workers produce the same serial (atomic `$inc`)
- [ ] Given 10,000+ bookings in a day, when the counter increments, then no collision occurs (monotonic, not random)
- [ ] Given sandbox validation, when a booking is submitted, then the barcode is accepted and echoed back in `valid_articles[].barcode_no`
- [ ] Given the counter is new, when no India Post carrier fields exist, then a migration adds the counter field with a safe initial value
