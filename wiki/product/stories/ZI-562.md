## User Story

As a **brand-new WooCommerce merchant who just installed WSS, has a legitimate WordPress Administrator role on her own store, and is paying for a plugin to connect Australia Post and ship her first orders this week**, I want **the plugin's "Let's Start Fulfilling" entry page to actually open when an Administrator clicks it — and, if something on my site genuinely makes my Admin user fail the plugin's permission check, to tell me which capability or option is missing and how to grant it from within WordPress**, so that **I am not bounced from a freshly installed paid plugin with a stock WordPress "Sorry, you are not allowed to access this page" error on day one, do not need to hand my Admin password to a third party over Zoom for an L3 ticket just to find out why my own plugin won't open, and the next merchant with a slightly non-default WordPress setup doesn't churn out of the trial during the same first click.**

---

## Acceptance Criteria (Simple)

- [ ] Store `glowrene.com.au` (account `ad4fa461-2435-45c9-aa99-ab89b81ea0a5`) — the user with the WordPress Administrator role — can open the "Let's Start Fulfilling" page without the "Sorry, you are not allowed to access this page" error, in both normal and Incognito sessions.
- [ ] WSS's permission check on the entry page accepts any user whose effective WordPress capabilities include the `manage_woocommerce` capability (or whichever capability is the documented requirement), rather than a stricter check that excludes legitimate Administrators on common Woo setups.
- [ ] When WSS denies a user, it replaces the generic WP "you are not allowed to access this page" page with a WSS-branded message naming the missing capability and linking to a one-page doc explaining how to grant it — instead of leaving the merchant to guess.
- [ ] A self-diagnostic appears on the WSS plugin row inside WP Admin → Plugins for any user with `manage_options`, showing whether the current user passes WSS's entry-page check and, if not, exactly which capability is missing — so support does not need a Zoom call and admin password to identify a permission-class fail.
- [ ] Onboarding telemetry counts "entry page denied" as a distinct install-blocker event and alerts the WSS team when more than one merchant in a 24-hour window hits it — so a regression cannot reach prod and lock out new installs unnoticed.

## Acceptance Criteria (Given/When/Then)

### Scenario: glowrene.com.au's Admin opens "Let's Start Fulfilling" after the fix

- **Given** the Administrator user on `glowrene.com.au` is logged in to WP Admin (Australia), with no role modifications beyond a default WooCommerce + Administrator setup, and WSS is installed and activated
- **When** that Admin clicks "Let's Start Fulfilling" from the WSS dashboard
- **Then** the carrier-setup page loads with no "Sorry, you are not allowed to access this page" error, in both a normal browser session and in Incognito
- **And** the Admin can complete the Australia Post connection inside the same session, without contacting PluginHive support or sharing admin credentials

### Scenario: Detect the same permission class on any fresh WSS install

- **Given** any WordPress site newly installs WSS, where the site's first Administrator user has WooCommerce installed and active
- **When** the Administrator opens the plugin for the first time
- **Then** the WSS entry page either loads successfully, or — if a documented capability is genuinely absent on that site — shows a WSS-branded denial page naming the missing capability and the exact WP doc that explains how to grant it
- **And** the merchant never sees the stock WordPress "Sorry, you are not allowed to access this page" message originating from the WSS plugin

### Scenario: Catch a future onboarding-blocker regression before customers churn

- **Given** WSS is in production and a code change tightens a capability check on any onboarding screen
- **When** that change reaches a real install and any user fails the new check
- **Then** an "entry page denied" telemetry event is recorded with the user's role, missing capability, store ID, and plugin version
- **And** a threshold alert fires (e.g. ≥1 in 24h on the latest release) so the WSS team sees the install-blocker the same day instead of via an L3 Zendesk + Zoom session 2 days into the trial
