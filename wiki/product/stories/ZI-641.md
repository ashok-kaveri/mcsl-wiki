## Problem Statement

On **FedEx REST** labels and the FedEx-generated **commercial invoice (ETD)**, the recipient street address is **merged into one line and printed twice** whenever Address Line 2 is populated. Reported by two merchants; the team marked it "fixed" three times and it kept recurring, because earlier fixes targeted the wrong layer.

- **Evidence**: Ticket #396998 — support visually confirmed *"'18 Hazel Grove Silsoe' does appear twice in the recipient address block on both the FedEx International Priority and FedEx Regional Economy labels"* for test order **UK3945**. Ticket #387062 — eurosec reported the same on orders 2424 / 2429 / 2437; reproduced internally on QA-MCSL order 1279.
- **Affected users**: ≥2 merchants directly; affects any FedEx REST order whose recipient address went through FedEx address validation/correction with both address lines filled.
- **Impact**: Wrong/garbled address on the shipping label and customs commercial invoice → risk of carrier rejection or misdelivery. Merchant-blocking.

## Root Cause (confirmed against the as-sent payload)

The package-view holds the actual FedEx REST request for UK3945 (order 8710317113608). The recipient sent to FedEx was:

```json
"recipient": {
  "address": { "city": "Bedford", "postalCode": "MK45 4GJ", "countryCode": "GB",
    "streetLines": ["18 Hazel Grove Silsoe", "18 Hazel Grove Silsoe"] },
  "contact": { "personName": "Justinas Zarnauskas", "companyName": "Kavaholikas MB" }
}
```

`recipient.address.streetLines` = **`["18 Hazel Grove Silsoe", "18 Hazel Grove Silsoe"]`** — Address Line 1 + Line 2 **merged** into one string and that string **duplicated**. Sender (`["V Nageviciaus 3"]`) and billing (`["kavoliuko"]`) carry one scheme each and are not affected. The contact name is single (personName + companyName are two distinct lines).

**Mechanism**: `buildStreetLinesArray` (`requestBuilder.js:152`) is called at `requestBuilder.js:193` with **both** address alias schemes — `(addressLine1, address_1, addressLine2, address_2, addressLine3, address_3)` — and only filters falsy/`--` values. The recipient (`order.shipping`) normally carries only `address_N`; but the FedEx **address validation/correction** path writes the validated street back as `addressLineN` (returned as a single merged token "18 Hazel Grove Silsoe") while the original `address_N` persists. So the merged value exists under **both** `addressLine1` and `address_1` → `buildStreetLinesArray` emits it twice.

> Note: the live order JSON looks clean (`address_1: "18 Hazel Grove"`, `address_2: "Silsoe"`, no `addressLineN`) because the package-view captured the request as it was *built*, before the order address was later re-cleaned. The package-view is ground truth.

Lineage: same `buildStreetLinesArray` call site introduced in commit `e8c8de4608` (PR #1862, 2024-05-17) — also the source of the related May 2026 FedEx REST address-correction defects.

## Fix

`storepepSAAS/server/src/shared/API/carriers/fedExRest/requestBuilder.js:193` — coalesce each physical line to a single alias source instead of passing both schemes:

```diff
- addressToReturn.streetLines = buildStreetLinesArray(addressLine1, address_1, addressLine2, address_2, addressLine3, address_3);
+ addressToReturn.streetLines = buildStreetLinesArray(addressLine1 || address_1, addressLine2 || address_2, addressLine3 || address_3);
```

Verified against the three schemes in the payload: recipient (only `address_N`) → `[address_1, address_2]` unchanged; sender (only `addressLineN`) → unchanged; broken recipient (both = merged) → `["18 Hazel Grove Silsoe"]`, single — **duplication gone**. Defensive alternative: `[...new Set(buildStreetLinesArray(...))]` to dedupe.

## Open Issues

## User Story

As a **merchant generating FedEx REST labels for orders with a second address line**,
I want **the recipient street address to print once, correctly**,
so that **my labels and customs commercial invoices are accepted by FedEx and my parcels are delivered to the right address**.

---

## Acceptance Criteria (Simple)

- [ ] Generating a FedEx REST label for an order with Address Line 1 + Line 2 populated produces `recipient.address.streetLines` with **one entry per physical line** (no duplicate, no merge-then-duplicate).
- [ ] UK3945 (and a fresh order with Line 2 populated) regenerated: street prints once on both the label and the commercial invoice.
- [ ] No regression for sender/ship-from (`addressLineN` scheme) or for recipients without a second line.
- [ ] Holds across FedEx International Priority and FedEx Regional Economy services.

## Acceptance Criteria (Given/When/Then)

### Scenario: Direct fix for reported issue
- **Given** a FedEx REST order whose recipient has Address Line 1 = "18 Hazel Grove" and Line 2 = "Silsoe" (and the address was validated/corrected)
- **When** a label is generated
- **Then** the FedEx request `recipient.address.streetLines` contains a single street entry (e.g. `["18 Hazel Grove Silsoe"]` or `["18 Hazel Grove", "Silsoe"]`)
- **And** the label and commercial invoice show the street exactly once.

### Scenario: Fix holds for similar configurations
- **Given** a different FedEx REST merchant with both address lines populated and address correction enabled
- **When** they generate a label
- **Then** the street is not duplicated, regardless of which alias scheme the stored address uses.

### Scenario: No regression on ship-from
- **Given** a ship-from address that uses the `addressLineN` scheme with a single line
- **When** a label is generated
- **Then** the shipper `streetLines` is unchanged (single, correct entry).
