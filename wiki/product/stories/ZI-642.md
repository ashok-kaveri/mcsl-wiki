## Problem Statement

When a Shopify order is cancelled, the MCSL app overwrites the StorePep order status to **`CANCELLED`** unconditionally — **even when a shipping label was already created** (status `LABEL_CREATED`, `SHIPPED`, or any post-label state). The order loses its post-label status and drops out of the merchant's label/fulfilment views, despite a real, already-purchased label and a valid tracking number.

- **Evidence**: Ticket #390510, comment 2026-06-30 (Keerthi, L3): *"the app is currently updating the status of 'Label Created' orders to Cancelled whenever the corresponding order is cancelled in Shopify, even though a shipping label has already been created. This is incorrect."*
- **Affected users**: Reported by loveofindia, but **not store-specific** — the cancel-flip is not gated by any feature toggle (see Root Cause), so it affects **any MCSL Shopify store** whose order has version tracking and is cancelled in Shopify after a label was created.
- **Impact**: Data integrity / order-state loss. Post-label orders silently leave their working views; merchant perceives the label/shipment as lost and cannot reconcile fulfilment. No customer-facing data exposure.

**Expected behavior**: once a label has been created, a later Shopify cancellation must **not** flip the StorePep order to `CANCELLED`. The order retains its post-label status. Cancellation should only apply when the StorePep order is in an early state — **`PROCESSING` or `INITIAL`** (per the merchant's stated requirement).

## Root Cause (confirmed against live code)

A Shopify cancellation arrives as a normal **order-update webhook carrying `cancelled_at`** (there is no dedicated cancel route). The cancel branch writes `CANCELLED` with **no check of the order's current status**, and it deliberately **bypasses** the status guard the normal update path uses. Trace (`storepepSAAS/server`):

1. **Cancellation detected** — `src/routes/internalOrderWebhook.js:51`
   `deriveStorePepStatus()`: `if (cancelledAt) return constants.CANCELLED;` (reads `order.cancelled_at`).

2. **Normal processing skipped, but the flip still runs** — `src/routes/internalOrderWebhook.js:70-79`
   ```js
   const status = deriveStorePepStatus(order, partialFulfilmentEnabled);            // = CANCELLED
   const shouldProcessOrder = (partialFulfilmentEnabled || orderStatusUpdateEnabled)
                              && status !== constants.CANCELLED;                     // = false
   const processedOrder = shouldProcessOrder ? await new OrderProcessingService(order).process() : order;
   ...
   const result = await updateOrder(shopFromDb, orderVersionDetails, processedOrder, newSocketEvent, status); // still called with CANCELLED
   ```
   Because `status === CANCELLED`, `shouldProcessOrder` is false — but `updateOrder(..., CANCELLED)` is still invoked at line 79, gated only by `shopFromDb && orderVersionDetails`. **No feature toggle protects this** (`orderStatusUpdateEnabled` / `partial.fulfilment.enabled` are bypassed).

3. **Cancel branch bypasses the normal guard** — `src/shared/orders/processOrderHelper.js:152`
   ```js
   if (type === constants.CANCELLED) {
     return processCanceledOrders(masterOrders, childOrders, { ... });   // early return
   }
   ```
   The normal update path (`processOrderImportForUpdatedOrder`) calls `checkIfAllSubOrdersAreInProcessing()` (`src/shared/API/stores/common/processOrders.js:104-125`, used at line 284) before applying an update — it silently drops updates on post-label/shipped orders. The cancel branch returns **before** ever reaching it.

4. **Unconditional flip** — `src/shared/orders/processCanceledOrders.js`
   ```js
   const orderFromDb = await db.subOrder.findOneWithLean({ subOrderUUID, storeUUID });   // :52
   const orderUpdateObject = {
     storePepStatus: constants.CANCELLED,        // :54  — written regardless of current status
     updatedBy: currentUser,
     lastOrderStatus: orderFromDb.storePepStatus, // :56  — current status captured but never checked
   };
   ```
   The current `storePepStatus` is read into `lastOrderStatus` (line 56) and then ignored. A `LABEL_CREATED`/`SHIPPED` sub-order is flipped to `CANCELLED`. The master order is then flipped at `processCanceledOrders.js:96-97` (gated only on all sub-orders flipping successfully), and tracking orders at line 76.

**Status constants** (`src/storePepConstants.js`): `INITIAL` (298), `PROCESSING` (353), `CANCELLED` (300), `LABEL_CREATED` (356), `SHIPPED` (366). Early/cancellable = `{INITIAL, PROCESSING}`; everything else is post-label.

## Provenance — old bug, recently exposed

- **Unconditional cancel write (the bug)**: `processCanceledOrders.js:53-59` dates to **2019-03** (commits `be59bd175` "order cancelled support added", `1beb23d65`, `dd769cd74`). The cancel path has never checked current status — ~7 years old.
- **Auto-trigger from Shopify cancel (the exposure)**: `deriveStorePepStatus` returning `CANCELLED` on `cancelled_at` and passing it into `updateOrder` is **2025-07-11, PR #2524 `feature/external_fulfilments`** (commit `3d950ad338`). The `/webhooks/orders/update/:storeUUID` auto-update route itself is **2025-03-17, PR #2279** (`810288891e`).
- **Takeaway**: the dangerous write is ancient; the mid-2025 order-status auto-update feature newly wired Shopify cancellations into it. That is why the symptom surfaces now. If loveofindia was recently enrolled in the auto-update / external-fulfilment feature, the timing matches. **Note for support**: disabling `orderStatusUpdateEnabled` for the merchant will **not** stop the wrong cancellations on its own — the cancel branch ignores that toggle; the code fix is required.

## Fix

`storepepSAAS/server/src/shared/orders/processCanceledOrders.js` — inside `markSubOrdersAsCanceledAndUpdateParticipatedBatches`, immediately after the sub-order is fetched (line 52), skip the flip when the current status is not an early state:

```diff
  const orderFromDb = await db.subOrder.findOneWithLean({ subOrderUUID, storeUUID });
+ if (![constants.PROCESSING, constants.INITIAL].includes(orderFromDb.storePepStatus)) {
+   // label already created (or shipped) — a Shopify cancellation must not override it
+   continue;   // do not flip to CANCELLED; do not count as an error
+ }
  const orderUpdateObject = {
    storePepStatus: constants.CANCELLED,
    updatedBy: currentUser,
    lastOrderStatus: orderFromDb.storePepStatus,
  };
```

This mirrors the existing `checkIfAllSubOrdersAreInProcessing` pattern (`processOrders.js:109,116` — `storePepStatus ∈ {PROCESSING, INITIAL, ...}`) but placed on the cancel path, where the original guard never runs.

**Self-consistency**: with the sub-order guard, a label-created order's sub-orders are skipped → `successCount !== subOrders.length` → `updateData.success` is false → the master-order flip (`processCanceledOrders.js:96-97`) and tracking-order flip are also skipped. Net result: the order correctly **retains** its `LABEL_CREATED`/post-label status.

**Open decisions for the dev team**:
- **Toggle vs global** — the team ships these changes behind per-account toggles (ZD comments 2026-06-13 / 06-17: "toggles are not enabled yet"; `orderStatusUpdateEnabledFor` exists). Decide whether the guard ships globally or behind a new toggle.
- **Mixed orders** — for an order with one `PROCESSING` + one `LABEL_CREATED` sub-order, the `PROCESSING` sub-order still cancels while the `LABEL_CREATED` one is preserved and the master is left un-cancelled. Confirm this partial outcome is desired.
- **Other early states** — `NOT_TO_SHIP` (and other non-shipped states) were **deliberately excluded** to match the stated requirement (`{PROCESSING, INITIAL}` only). Revisit if never-shipped orders should also be cancellable.

## User Story

As a **merchant who has already generated and paid for a shipping label**,
I want **a later cancellation of that order in Shopify to leave the StorePep order in its label-created state**,
so that **I don't lose track of real, dispatched shipments and don't have to reconcile "vanished" labelled orders by hand**.

---

## Acceptance Criteria (Simple)

- [ ] Cancelling a Shopify order whose StorePep order is `LABEL_CREATED`/`SHIPPED` leaves the StorePep status unchanged (order keeps its post-label state and stays in its views).
- [ ] Cancelling a Shopify order whose StorePep order is `PROCESSING` or `INITIAL` still marks it `CANCELLED` (existing behavior preserved).
- [ ] The master order is only marked `CANCELLED` when all its sub-orders were eligible to cancel.
- [ ] No regression for other MCSL Shopify stores using order status auto-update.

## Acceptance Criteria (Given/When/Then)

### Scenario: Label-created order is protected
- **Given** a StorePep order in `LABEL_CREATED` state (label generated, tracking valid)
- **When** the merchant cancels that order in Shopify and the cancellation webhook is processed
- **Then** the StorePep order (sub-orders + master) retains `LABEL_CREATED`
- **And** it remains visible in the label/fulfilment views, not in Cancelled.

### Scenario: Early-state order still cancels
- **Given** a StorePep order in `PROCESSING` (or `INITIAL`) with no label created
- **When** the merchant cancels that order in Shopify
- **Then** the StorePep order is marked `CANCELLED` as before.

### Scenario: Mixed order
- **Given** an order with one `PROCESSING` sub-order and one `LABEL_CREATED` sub-order
- **When** the order is cancelled in Shopify
- **Then** the `PROCESSING` sub-order is cancelled, the `LABEL_CREATED` sub-order is preserved
- **And** the master order is not marked `CANCELLED`.
