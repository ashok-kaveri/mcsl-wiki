## User Story

As a **PluginHive product manager triaging new-carrier requests where I cannot start any feasibility work until I have carrier-side API docs and a technical contact**,
I want **the new-carrier-request intake to capture carrier API documentation and a technical contact up front — or give the customer a clear self-serve path to obtain them — before the request enters the dev queue**,
so that **feasibility scoping starts immediately instead of stalling for weeks while support chases docs by email, and small-plan requests like TryParcel don't consume disproportionate support time before we even know if integration is possible**.

---

## Acceptance Criteria (Simple)

- [ ] Siam Computing (ticket #384207) has shared TryParcel's API documentation URL and a named technical/account contact, captured in the request record
- [ ] The in-app "Request New Carrier" form requires (or strongly prompts) API docs URL + carrier contact before submission, and explains why these are needed
- [ ] Support agents have a one-click template to send to customers requesting carrier docs/contacts, referencing this request pattern
- [ ] When carrier docs arrive, they are attached to the request record so any dev picking up the scope can find them without re-asking support
- [ ] A "ready for scoping" flag is set automatically once docs + contact are present; product team sees a sorted queue of ready-to-scope requests

## Acceptance Criteria (Given/When/Then)

### Scenario: Collect TryParcel docs from Siam Computing
- **Given** the TryParcel request is logged against ticket #384207 with the customer asked to provide contacts
- **When** Siam shares an API doc URL or technical contact in a reply
- **Then** the carrier request record is updated with the docs/contact, and the request is marked "ready for scoping"
- **And** the product team is notified the request is unblocked

### Scenario: Prevent future requests from missing prerequisites
- **Given** any merchant opens the "Request New Carrier" form in-app
- **When** they fill it out
- **Then** the form explicitly asks for carrier URL, API docs URL, and technical contact, with help text explaining these accelerate the scoping process
- **And** if the customer doesn't have these, the form presents a prewritten email template the customer can forward to their carrier rep

### Scenario: Scale triage across many carrier requests
- **Given** multiple new-carrier requests exist in the system
- **When** a product-team member opens the new-carrier dashboard
- **Then** requests are sorted into "ready for scoping" (have docs + contact) vs "waiting for docs" (don't)
- **And** each request shows: requesting customer plan tier, merchant volume signal, and days-waiting — so PM can prioritize
