## User Story

As a **developer onboarding a new Canada-based WooCommerce merchant (Ultraseat) with a FedEx-visible partnership thread**,
I want **the plugin's FedEx registration UI to either succeed or tell me exactly which field doesn't match FedEx's portal data**,
so that **my new customer can self-serve their carrier setup in minutes instead of sending screenshots, FedEx invoices, and waiting days for the PluginHive team to manually add the account while our FedEx partner watches the CC thread**.

---

## Acceptance Criteria (Simple)

- [ ] Kelly's FedEx account for Ultraseat is registered and active in the WSS plugin without further manual PH-side intervention.
- [ ] The plugin-side FedEx registration UI surfaces a specific actionable error (e.g. "Account number does not match the address on file at FedEx") instead of a generic failure.
- [ ] A diagnostic log captures the exact FedEx API response at registration time so support can root-cause future UI failures without asking for invoices.
- [ ] The root cause of Kelly's specific UI-side registration error is documented in a known-issues/runbook entry visible to support.
- [ ] A merchant in a cross-time-zone geography can complete FedEx registration without a live call with PluginHive support.

---

## Acceptance Criteria (Given/When/Then)

### Scenario: Ultraseat registration completes
- **Given** Kelly has provided FedEx account details and a recent FedEx invoice for Ultraseat
- **When** PluginHive applies the registration via the same code path the plugin UI uses
- **Then** the account is added, visible in Ultraseat's carrier list, and labels can be rate-shopped
- **And** Kelly receives a public update confirming the account is live

### Scenario: Any merchant hitting the same UI error gets actionable feedback
- **Given** a new WooCommerce merchant enters FedEx credentials that don't match FedEx's portal record
- **When** they click Register in the plugin UI
- **Then** the UI shows the specific mismatching field (address, zip, account number, contact name) as returned by FedEx
- **And** the raw FedEx registration API response is captured in support logs tagged with the account ID

### Scenario: Support can root-cause without invoices
- **Given** a new registration failure is reported by any customer
- **When** support opens the ticket in Zendesk
- **Then** the linked diagnostic log shows the FedEx API error code and message verbatim
- **And** support can match it to a documented known-issue entry with a resolution path that does not require the customer to email invoice PDFs
