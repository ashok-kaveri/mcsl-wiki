## Open Issues

1. **Canada Post manifest generated on label date, not fulfillment date** — Manifest groups all pending labels into next fulfillment batch, not filtered by selected orders. Customer's workflow: generate label days in advance, fulfill when ready to ship. Current plugin behavior includes unpacked orders in manifest sent to Canada Post, creating "ghost packages" when manifest doesn't match actual shipments being picked up. Blocked: Development (credential/testing setup). Severity: High — customer reports this "messes with workflow drastically" and creates unreliable manifests causing logistics friction with Canada Post. Area: carrier-config. (Comment #15, #20, #22)

2. **Label cancellation workflow broken for orders in Label-Created state** — Customer needs to change label after creation (e.g., product out of stock discovered during packing). Suggested workaround (mark fulfilled → change to processing → cancel shipment → regenerate label) fails; label cannot be cancelled via API because Canada Post no longer supports cancellations (labels created after 2025-07-02 can only be discarded, not cancelled through API). Blocked: Customer (Canada Post API limitation + plugin limitation). Severity: High — forces manual shipping outside plugin. Area: label-generation. (Comment #12, #39, #40)

3. **Automatic carrier fallback when primary carrier fails** — Customer selected Canada Post for order #125764 but plugin auto-selected CANPAR when Canada Post request failed. Customer requires explicit carrier selection with user override, not automatic fallback. Blocked: Development (feature design). Severity: Medium — causes incorrect carrier selection and shipping delays. Area: carrier-config. (Comment #41, #42, #44)

## Resolved Issues

1. **Understanding of manifest timing** — Explained and confirmed that manifests are generated by Canada Post API in next fulfillment batch after label creation, not on fulfillment action date. (Comment #4, #15)

  - Label cancellation impossible with current Canada Post API limits
  - Automatic carrier fallback breaks shipping workflow
  - Customer describes plugin as "unreliable" with "tons of issues" incompatible with their business model

## User Story

As a **Canadian artisan food producer who creates shipping labels 2-3 days before packing — because my workflow requires inspecting each jar of jam before boxing — and who has spent 137 days reporting that Canada Post manifests include unpacked orders as "ghost packages" that the carrier scans but can't find at pickup**,
I want **manifest generation to include only orders I've marked as fulfilled, so the manifest matches the actual boxes on my loading dock and Canada Post stops flagging phantom entries every morning**,
so that **my daily manifest is accurate from the first print, I don't spend 20 minutes every morning reconciling ghost packages with the courier, and Canada Post stops questioning the reliability of my shipments**.

## Acceptance Criteria (Simple)

- [ ] Manifest generation includes only orders marked as fulfilled — labels in "Label Created" (unfulfilled) status are excluded
- [ ] Manifest includes a "Fulfillment Date" column so the merchant can verify timing
- [ ] Orders fulfilled across multiple days appear on the manifest for the day they were fulfilled, not the day the label was created
- [ ] Greaves Jams & Marmalades confirms manifests match actual packed shipments for 5 consecutive business days
- [ ] No regression for merchants who create and fulfill labels same-day (existing workflow unchanged)

## Acceptance Criteria (Given/When/Then)

### Scenario: Unfulfilled labels excluded from manifest
- **Given** 10 labels created today but only 6 orders fulfilled (4 still being packed)
- **When** the merchant generates today's Canada Post manifest
- **Then** the manifest contains only the 6 fulfilled orders
- **And** the 4 unfulfilled orders appear on the next manifest after they're fulfilled

### Scenario: Multi-day label-to-fulfillment workflow
- **Given** labels created Monday for 20 orders, 12 fulfilled Monday and 8 fulfilled Tuesday
- **When** Tuesday's manifest is generated
- **Then** it includes the 8 orders fulfilled Tuesday (not all 20 from Monday's label batch)
- **And** Monday's manifest already captured the 12 that were fulfilled Monday

### Scenario: Same-day workflow unaffected
- **Given** a merchant who creates labels and fulfills orders within the same hour
- **When** they generate a manifest at end of day
- **Then** all same-day fulfilled orders appear on the manifest as before
- **And** behavior is identical to pre-change for this workflow
