## Open Issues

1. **Canada Post manifest auto-generates for ALL pending labels at the next fulfilment batch, not for operator-selected orders** — Customer flow generates labels in advance (paperwork + label first, then packing team prepares, then operator marks fulfilled when ready). WSS currently manifests every pending label on the next batch trigger, so labels that haven't actually been packed/shipped end up on a manifest. Customer's concern: this creates an "unreliable manifest" — Canada Post may auto-scan and create "ghost packages" in their tracking system, causing customer-service grief. Engineering completed an "fulfill + transmit only selected orders" enhancement on 2026-02-24, but release is blocked: testing required Canada Post account credentials, customer agreed to credentials-for-test-on-her-site but declined sharing login credentials for PluginHive's test environment. Blocked: engineering (need a credential-free testing path). Severity: high (paying Yearly Popular customer who explicitly stated the plugin is "unreliable" for her workflow; 4-month-old dev_needed item; "Planned for 378"). Area: label-generation. (Comments #1, #5, #6, #10, #15, #23, #26, #36, #46)

2. **Label cancellation is broken after Canada Post's July 2, 2025 policy change — and the plugin doesn't surface a "discard label" path** — Canada Post no longer allows label cancellation via API (per their chat-bot quote shared by customer: labels can simply be discarded; charges only occur after shipment induction). The plugin still tries to use the cancellation API and fails, leaving the order in a state where a new label can't be generated. Customer's requested capability: a way to mark an existing label as "invalid" / "do not ship" and generate a new one without going outside the plugin (NOT allowing two active labels simultaneously — explicit guard against double-shipment). Workarounds offered by support didn't work in practice (Comment #40). Blocked: engineering (needs explicit "discard label and regenerate" UX for the post-2025-07-02 Canada Post regime). Severity: high (forces shipments outside the plugin every time, defeating the purpose of the subscription for affected orders). Area: label-generation. (Comments #10, #13, #14, #39, #40, #41)

3. **Plugin auto-selects an alternate carrier when the chosen one fails — no opt-out** — When the selected carrier service fails due to a technical or carrier-side error, the plugin silently substitutes the next available service. Customer reported a real instance (order #125764, 2026-03-09): a Canada Post order generated a Canpar label without her consent. Per Manohar (Comment #43): the auto-fallback is by design. Customer's demand: this behavior must be opt-in, not the default — she needs full control over carrier selection (with manual fallback when the primary fails, not automatic). Blocked: engineering (new setting + behavior change for failed-service fallback). Severity: high (customer relationships with carriers are commercial; auto-switching changes contractual cost and brand experience without merchant consent). Area: rate-shopping. (Comments #42, #43, #44, #45)

## User Story

As a **merchant with commercial contracts at specific carriers** (Canada Post Contract account for the cheaper rates, Canpar only as a secondary), I want **the plugin to NEVER silently substitute an alternate carrier when my chosen carrier's API fails — instead surface the error to me so I can decide what to do**, with auto-fallback as an explicit opt-in setting, so that **I don't end up with surprise Canpar labels on orders I priced for Canada Post (changing my landed cost and customer brand experience), and I can trust that the carrier I selected is the carrier that actually ships the parcel**.

---

## Acceptance Criteria (Simple)

- [ ] Add a setting 'On chosen-carrier failure: surface error / auto-fallback to next available' with the default being SURFACE ERROR (not silent substitution)
- [ ] When the setting is on 'surface error', any chosen-carrier API failure stops label generation for that order and presents the error message + retry/change-carrier options
- [ ] When the setting is on 'auto-fallback' (opt-in), the current behavior is preserved and labeled clearly in the UI so the merchant knows it's happening
- [ ] Greaves Jams order #125764 type incidents (Canpar label on a Canada Post order) cannot recur with default settings post-fix
- [ ] Documentation explains the trade-off so merchants make an informed choice: convenience (auto-fallback) vs carrier-contract-integrity (surface-error default)

---

## Acceptance Criteria (Given/When/Then)

### Scenario: Default behavior: chosen-carrier failure surfaces an error, no substitution

- **Given** a merchant with the default setting (surface error)
- **When** the chosen Canada Post service fails due to a transient carrier-side API error during label generation
- **Then** the order stays in pre-label state with a clear error explaining the carrier failure and offering retry or change-carrier
- **And** no Canpar (or other alternate) label is silently generated

### Scenario: Opt-in auto-fallback works as before, with visible signal

- **Given** a merchant who has explicitly enabled auto-fallback in settings
- **When** the chosen carrier fails and the plugin selects the next available
- **Then** the label IS generated on the alternate carrier (preserving current behavior), AND the plugin clearly marks the order as 'auto-fallback occurred: <original> → <alternate>'
- **And** the merchant sees the fallback in a visible audit log without having to dig

### Scenario: Bulk batch with one chosen-carrier failure does not silently mix carriers

- **Given** a batch of 20 Canada Post orders processed together
- **When** order #15 hits a Canada Post API failure
- **Then** orders #1-14 generate Canada Post labels, order #15 stops with a clear error, orders #16-20 continue with Canada Post
- **And** no order in the batch silently receives a Canpar label
