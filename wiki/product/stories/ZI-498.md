## User Story

As a **Canadian merchant whose customer-facing vendor dashboard lives on my own domain but ends in '/pluginhive'**, I want **the trailing path segment of the dashboard URL configurable to match my own brand naming**, so that **my customers don't see a third-party vendor name in URLs on my store and the dashboard feels like a native part of my site rather than a bolted-on tool**.

---

## Acceptance Criteria (Simple)

- [ ] Merchant can configure a custom dashboard slug (e.g., '/dashboard/shipping') from app settings without contacting support
- [ ] Existing /pluginhive URLs continue to work (redirect or alias) so customers with bookmarks are not broken
- [ ] Slug validation prevents conflicts with existing WordPress routes and reserved paths (e.g., /admin, /wp-content)
- [ ] Customer somtico.com can rename their dashboard URL trailing segment in a single setting change
- [ ] Documentation explains the trade-off between custom slug branding and the redirect-from-old-slug behavior

---

## Acceptance Criteria (Given/When/Then)

### Scenario: Merchant sets a custom dashboard slug

- **Given** a WSS install with dashboard at https://somtico.com/dashboard/pluginhive/
- **When** the merchant changes the trailing slug to 'shipping' in app settings
- **Then** the dashboard becomes reachable at https://somtico.com/dashboard/shipping/ for all users
- **And** requests to the old /pluginhive path redirect (HTTP 301) to /shipping without breaking existing customer bookmarks

### Scenario: Reserved or conflicting slugs are rejected with a clear error

- **Given** the merchant attempts to set the slug to /wp-admin or /admin
- **When** they save the slug-customization settings form and submit it to the server
- **Then** validation rejects the change with a message naming the reason ('reserved path') rather than failing silently or breaking the site
- **And** the original slug is preserved unchanged

### Scenario: Default slug remains for stores that don't customize

- **Given** an existing WSS install that has never customized the slug
- **When** they upgrade to the version that adds the setting
- **Then** the default '/pluginhive' slug continues to work exactly as before
- **And** no migration or customer action is required for the upgrade
