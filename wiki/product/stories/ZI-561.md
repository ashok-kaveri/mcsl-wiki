## User Story

As a **WSS merchant shipping internationally on FedEx who pays for the plugin to ship orders without becoming a postal-code archaeologist**, I want **a FedEx address rejection to tell me, before I ever click "generate label", exactly which field on the destination is invalid and what FedEx will accept — instead of letting me click the button, fail with the cryptic `Destination postal code missing or invalid`, and then asking me to "go cross-check on the FedEx website"**, so that **a single bad-postcode order does not silently turn into a Zendesk ticket and a one-day delay for one of my customers, and so I can fix or override the address inside WSS in the same minute I spotted it instead of context-switching to FedEx's portal to diagnose my own shipping system.**

---

## Acceptance Criteria (Simple)

- [ ] Order #258685 on leyakoreya.ee (LEYAKO OU, account `512d0981-b53e-4eea-8c51-b9fed8788975`) is either re-attempted successfully after the destination postal code is corrected, or shows a structured "invalid postal code for destination country `X`" message at the order level — not a generic carrier failure modal.
- [ ] FedEx address rejections referencing destination postal code, postal-code-mismatch-with-country, or unsupported postal code zones are mapped to a human-readable, field-pointing error in WSS — naming the offending field and the value the carrier saw.
- [ ] Where FedEx publishes a postal-code format/length per destination country (e.g. NL `1234 AB`, US `NNNNN`), the WSS order edit screen validates the destination postal code against that pattern at save time, so the error surfaces during data entry rather than at label time.
- [ ] The destination-address sanity check used at label time is also exposed as a "Pre-flight address" button on the order, so a merchant can audit problematic orders in bulk before the warehouse hits "print all".
- [ ] When a postal-code-class rejection happens, the merchant gets a one-click "Edit destination" link straight from the error toast into the WooCommerce order address — no FedEx-portal round trip required.

## Acceptance Criteria (Given/When/Then)

### Scenario: LEYAKO retries Order #258685 with a corrected postal code

- **Given** Order #258685 has a destination postal code that FedEx rejects as "missing or invalid" (currently surfacing as a generic label failure on leyakoreya.ee)
- **When** the LEYAKO operator opens the order in WSS and clicks "Generate FedEx label"
- **Then** instead of a generic failure, the order shows a "Destination postal code is invalid for destination country `<CC>` — FedEx expects format `<pattern>`" inline error, with an "Edit destination address" link
- **And** correcting the postal code to a value matching the country's format allows the same operator to generate the label successfully without contacting support

### Scenario: A different WSS merchant on FedEx hits an unsupported postal code on a new order

- **Given** a WSS merchant on FedEx receives a WooCommerce order with a destination postal code that does not match the country's postal-code format (e.g. a 4-digit code in a country expecting 5 digits)
- **When** the order is saved in WooCommerce, before any label attempt
- **Then** the merchant sees a yellow banner on the order — "Destination postal code does not match expected format for `<country>`. FedEx will reject this label." — with a one-click address edit
- **And** the FedEx label flow does not attempt the call until either the postcode is corrected or the merchant explicitly chooses "Submit anyway"

### Scenario: Stop "go check FedEx's website" as a support reply

- **Given** any merchant on WSS receives a carrier-side address rejection (postal code, missing state, unsupported country, etc.)
- **When** the rejection is rendered in the app
- **Then** the error message names the field, shows the value as the carrier received it, and lists the carrier's expected format — sourced from a maintained mapping shipped with the app
- **And** support no longer has to advise the merchant to recreate the shipment on the carrier's portal to identify which field is wrong — they can see it inline in WSS
