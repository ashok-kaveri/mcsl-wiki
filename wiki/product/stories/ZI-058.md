## User Story

As a **high-volume Australian merchant shipping ~500 eParcel labels per month on the $29 plan**,
I want **bulk label generation to complete in seconds instead of 8+ seconds per label**,
so that **my fulfilment team isn't standing idle waiting for labels while the same-day cutoff approaches, and I don't have to question whether the $29 plan is worth paying for an app that's slower than generating labels by hand on the AuPost portal**.

---

## Acceptance Criteria (Simple)

- [ ] Bulk eParcel label generation for 11MODE (95a96e-3.myshopify.com) completes in under 3 seconds per label, measured across a 10-order batch
- [ ] Latency is reproducible in QA using the customer's specific order IDs or at least one comparable AU eParcel account
- [ ] Root cause (carrier API call, DB query, rate lookup, sync side-effect) is identified and documented — not just "it's faster now"
- [ ] A performance regression test locks in the target latency for eParcel bulk runs so this doesn't silently creep back up
- [ ] Customer is proactively informed of the fix with before/after timings and apology for the 22-day SLA breach

---

## Acceptance Criteria (Given/When/Then)

### Scenario 1: 11MODE bulk batch completes in seconds
- **Given** Matt at 11MODE selects 10 AuPost eParcel orders from his order grid
- **When** he clicks "Generate Labels" in bulk
- **Then** all 10 labels are generated in under 30 seconds total (p95)
- **And** progress is visible per-order so he knows the process is alive, not hung

### Scenario 2: Fix applies to other AU eParcel customers, not just 11MODE
- **Given** any AU merchant with AuPost eParcel configured and 10+ pending orders
- **When** they run a bulk label generation
- **Then** per-label latency stays under the same 3-second target
- **And** no carrier-specific slow path exists for eParcel vs other AuPost services

### Scenario 3: Latency regressions are caught before customers report them
- **Given** the performance regression test suite runs on every carrier-integration change
- **When** bulk eParcel generation exceeds the 3-second-per-label budget
- **Then** the test fails in CI before the change merges to production
- **And** the alert points to the specific code path that slowed down
