## User Story

As an **agency dev whose end client sells custom promotional bunting by the metre and therefore cannot use weight-plus-dimension box packing the way TNT's API expects**, I want **the plugin to return TNT Overnight Express rates that match the carrier's own calculator within a few cents — not roughly half the real price, where Road Express matches but Overnight is off by $128 on a single shipment**, so that **my client doesn't absorb $128+ per overnight order in undercharged shipping or have to abandon TNT entirely in favour of generating labels manually on TNT's portal because my plugin's price doesn't match the price they will be billed.**

---

## Acceptance Criteria (Simple)

- [ ] On a checkout where Road Express returns $79.30 (matching TNT's calculator), Overnight Express returns within ±5% of TNT's calculator value of $232.41 for the same lane, weight, and origin/destination postcodes.
- [ ] When packaging is set to Weight-Based (no per-product dimensions) and the carrier requires dimensions to price a service, the plugin either (a) imputes a defensible per-product dimension from the weight + a configurable density / box mapping, or (b) returns the rate as "unavailable for this packaging method" with an explicit reason — never silently returns a discounted-looking rate that the carrier will not honour.
- [ ] The agency dev can run a single-shipment "rate explainer" that shows, for each service returned, what weight and what dimensions the plugin sent to TNT's API and what the carrier returned — so a $103.61 vs $232.41 gap is debuggable without filing a ticket.
- [ ] The post-call guidance path — pack into a `52x36x35` box with a derived per-metre height like `0.07 cm` — is documented in-plugin (not just on a call recording) and can be applied to a metre-sold product line in under 15 minutes.
- [ ] About Bunting's Custom Promotional Bunting line, when configured with the documented per-metre dimension formula, returns Overnight Express rates within ±5% of TNT's calculator for at least three test lanes (Sydney metro, regional NSW like 3460 Basalt, and one interstate VIC).

## Acceptance Criteria (Given/When/Then)

### Scenario: Overnight Express rate matches the carrier calculator after the dimension-derivation fix

- **Given** About Bunting has applied the post-call guidance — pack "Custom Promotional Bunting" into a `52x36x35` box with a per-metre height derived from the carrier's calculator (e.g. `0.07 cm` per metre) — and the plugin is configured with TNT Australia using Weight-Based Packing
- **When** Jarrod adds 10 metres of Custom Promotional Bunting to the cart for a Sydney-metro overnight lane and TNT's own calculator returns $232.41 for Overnight Express
- **Then** the plugin returns TNT Overnight Express at $232.41 ±5% at checkout in the same currency, and TNT Road Express continues to return $79.30 ±5%
- **And** the plugin's rate-debug log shows the exact weight (kg), exact dimensions (cm) and exact carrier service code sent to TNT's API for that quote — so the value can be verified manually

### Scenario: Plugin refuses to silently undervalue rates when dimensions are unknown

- **Given** a WSS merchant has configured Weight-Based Packing for a product line whose products do not carry per-product dimensions and the chosen carrier service requires dimensions to price (e.g. TNT Overnight Express, FedEx Priority)
- **When** the checkout requests a rate for that lane
- **Then** the plugin returns the service as "rate unavailable — service requires per-product dimensions; switch to Boxed Packing or set dimensions per variation" rather than returning a rate that is roughly half of the carrier's true price
- **And** the merchant admin shows a one-time setup warning listing every carrier service in the merchant's account that is currently being skipped for this reason, so it cannot be missed

### Scenario: Self-serve rate explainer prevents a third L3 escalation on the same store

- **Given** a WSS merchant who suspects a returned rate does not match the carrier's own calculator
- **When** the merchant clicks "Explain this rate" next to a rate returned at checkout (or in an admin preview)
- **Then** the explainer shows side-by-side: weight sent, dimensions sent, carrier service code, raw carrier API response, and a deep link to the carrier's own calculator pre-filled with the same values
- **And** if the calculator price differs by more than 5%, the explainer surfaces the likely cause (missing dimension, wrong packaging method, fuel-surcharge handling, account-rate vs published-rate mismatch) with a doc link — replacing the "send us a Loom video" support cycle on this ticket
