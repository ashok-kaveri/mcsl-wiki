## User Story

As a **Japanese D2C merchant (Cybershaft) on MCSL shipping into APAC where Hong Kong and Taiwan addresses simply do not have postal codes — and where my FedEx REST migration just turned every HK/Taiwan rate quote into a `postalCode can not be null` rejection**,
I want **MCSL to handle the absence of a destination postal code as a normal case for the small list of countries that don't use postal codes (HK SAR, Taiwan, Ireland, etc.), inject FedEx's documented placeholder (`000000`) into the REST rate/label request transparently, and produce a working rate and label like it does for every other destination**,
so that **my checkout doesn't quietly stop quoting FedEx rates to my Hong Kong customers (who then abandon to a competitor), I stop the manual DHL+FedEx dual-routing dance I'm doing as a workaround, and Morrison in Belgium — who's been waiting 11 days on the same fix on the FedEx app — gets unblocked at the same time because the underlying logic is shared**.

## Acceptance Criteria (Simple)

- [ ] MCSL maintains a server-side list of "no-postal-code" destination countries (HK SAR, Taiwan, Ireland, Panama, and any others per ISO postal-code data) and, for FedEx REST rate/label requests destined to those countries, injects FedEx's documented placeholder (`000000` or whatever FedEx's spec requires) into the postal-code node — instead of omitting the node and letting FedEx reject
- [ ] Cybershaft's exact failing scenario (Japan origin → Hong Kong destination, no postal code at checkout) now returns FedEx rates and produces a working label, verified end-to-end on a real test order
- [ ] The same fix is verified against Taiwan as well, and against at least one other ISO no-postal-code country (e.g. Ireland) to prove the list-driven approach generalises beyond what one ticket reported
- [ ] No regression for destinations that DO use postal codes (China with a real pincode, US ZIP, UK postcode) — the placeholder only fires for countries on the no-postal-code list, never as a fallback for "merchant didn't enter a postal code on an address that should have one"
- [ ] Coordinated release: the same fix lands in both the MCSL code path (this card) and the standalone FedEx Shopify app code path (ZI-548 on ticket #387886) in the same release window — Morrison (Belgium) and Cybershaft (Japan) both stop being blocked on the same day

## Acceptance Criteria (Given/When/Then)

### Scenario: Cybershaft generates a FedEx rate to Hong Kong without entering a postal code
- **Given** Cybershaft has an unfulfilled Shopify order shipping from Japan to a Hong Kong address with no postal code in the Shopify order (because HK doesn't use them)
- **When** the packer opens the rate panel on that order in MCSL and FedEx REST is the configured carrier
- **Then** MCSL automatically injects FedEx's `000000` placeholder into the REST request, FedEx returns rate options, the panel shows FedEx services without a "postalCode can not be null" error, and label generation produces a working FedEx label with a real tracking number

### Scenario: Other no-postal-code destinations work for any MCSL merchant, not just Cybershaft
- **Given** any MCSL merchant on FedEx REST quoting to a destination country in the no-postal-code list (Hong Kong, Taiwan, Ireland, Panama, etc.) with no postal code on the address
- **When** they request a rate via the rate panel or the rate API
- **Then** FedEx returns valid rates without error, the resulting label prints correctly, and the merchant did not have to manually edit Shopify to insert a fake postal code first

### Scenario: A real US/UK/JP order with a missing postal code still fails loudly (no quiet fallback)
- **Given** an MCSL merchant has a Shopify order to a US, UK, or Japan destination where the postal code IS used by the country but happens to be empty on the order (data quality bug at checkout)
- **When** they try to generate the FedEx label
- **Then** MCSL does NOT inject the placeholder (because the destination country uses postal codes) — instead it surfaces a clear "Postal code required for destination <country>" error so the merchant can fix the Shopify order, exactly as today

### Scenario: Coordinated cutover with the FedEx app fixes Morrison and Cybershaft together
- **Given** both this card (MCSL) and ZI-548 on ticket #387886 (FedEx app) target the same June release window
- **When** the coordinated release lands in production for both apps and the customers retry their previously-failing flows
- **Then** Cybershaft's HK rate quote and Morrison's HK label generation both work the same day, both customers are proactively contacted with the release notes, and Morrison's "this is what they told me LAST WEEK" follow-up does not recur because the ETA committed on 2026-05-18 was actually held
