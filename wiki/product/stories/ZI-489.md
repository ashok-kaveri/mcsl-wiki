## User Story

As a **merchant who selects FedEx Express Saver at checkout with custom packaging configured in my store**,
I want **label generation to succeed without "Packaging type not allowed" errors when FedEx validates the request**,
so that **I don't have to manually regenerate failed labels, customers get their tracking numbers immediately after checkout, and I'm not losing orders because the app is sending invalid special service codes (like "One Rate") that FedEx rejects for my packaging type**.

---

## Acceptance Criteria (Simple)

- [ ] Labels generate successfully for FedEx Express Saver service with custom packaging without "Packaging type not allowed" errors
- [ ] FedEx REST API label requests do NOT include "One Rate" special service unless merchant explicitly configured One Rate packaging
- [ ] Pierre (grapebottle.myshopify.com) can generate labels for order #9866 and similar orders without manual intervention
- [ ] Label generation logs clearly show which special services are being sent to FedEx API and why
- [ ] Packaging type validation errors are caught before API submission with user-friendly error messages

---

## Acceptance Criteria (Given/When/Then)

### Scenario: Custom packaging with FedEx Express Saver generates label successfully (Happy path)
- **Given** a merchant has configured custom packaging (not FedEx One Rate) for their products
- **And** a customer selects FedEx Express Saver service at checkout
- **When** the merchant generates a label for that order
- **Then** the label generates successfully without "Packaging type not allowed" error
- **And** the FedEx REST API request does NOT include "One Rate" special service
- **And** the label shows correct packaging type matching merchant configuration

### Scenario: One Rate special service only sent when merchant uses One Rate packaging (Broader fix)
- **Given** multiple merchants with different packaging configurations (custom, standard, One Rate)
- **When** labels are generated via FedEx REST API
- **Then** "One Rate" special service is ONLY included for merchants with One Rate packaging configured
- **And** custom packaging merchants never receive "Packaging type not allowed" errors
- **And** standard packaging merchants generate labels with correct package type codes

### Scenario: Validation catches packaging/service mismatches before API submission (Prevention)
- **Given** a merchant attempts to generate a label with incompatible packaging and service combinations
- **When** label generation is triggered
- **Then** client-side validation detects the packaging/service mismatch BEFORE calling FedEx API
- **And** merchant sees a clear error message: "FedEx Express Saver is not compatible with One Rate packaging. Please select a different service or change packaging type."
- **And** merchant can self-correct the issue without failed API calls or support tickets
