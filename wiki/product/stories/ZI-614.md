# Ticket #393559 — FedEx Freight LTL Rate Failure for Orders with 15+ Line Items After REST Migration

- **Customer**: Amy (2af7c3-29.myshopify.com)
- **Product**: Shopify Multi Carrier Shipping Label
- **Duration**: 2026-06-03 to 2026-06-18 (16 days, ongoing)

## Timeline & Key Phases

### Phase 1: Initial Rate Failure Report (2026-06-03)
- Customer reported shipping rates not pulling through for order #3680 (551.69 lbs)
- Support identified FedEx Freight LTL services (Economy + Priority) configured via automation rules for shipments over 150 lbs
- Error: "The combination of freight line items and shipment special services exceeded limit"
- Support advised removing special services (Liftgate Delivery, Call Before Delivery) and reprocessing

### Phase 2: Workaround Attempted, Root Cause Escalated (2026-06-03 to 2026-06-04)
- Customer removed special services but reprocessing did not resolve the issue — problem persisted across multiple orders
- L3 escalation raised: root cause identified as two distinct issues:
  1. Line items with weight < 0.5 lbs failing with `LINEITEM.WEIGHT.REQUIRED` error
  2. FedEx REST API limit of 15 unique line items per Freight request being hit

### Phase 3: Partial Fix Deployed, Enhancement Pending (2026-06-04)
- Fix deployed: app now automatically sets minimum 0.5 lb weight for products below threshold
- 15 line-item limit confirmed as a FedEx REST API constraint (potentially a regression from FedEx Web Services → REST migration)
- Customer (Amy) rejected the partial fulfillment workaround due to DaVinci WMS integration dependency
- Customer explicitly waiting for the full fix; no ETA provided at time of response

### Phase 4: ETA Provided (2026-06-18)
- Engineering (Veena) provided ETA: on or before 7th July for the line-item splitting enhancement (Comment #16)

## Open Issues

1. **FedEx Freight Orders with 15+ Unique Line Items Fail to Fetch Rates** — FedEx REST API enforces a 15 unique line-item limit per Freight rate request. The app does not currently split oversized orders into multiple sub-requests, causing rate failures for orders with more complex product mixes. Customer cannot use partial fulfillment workaround due to DaVinci WMS integration. Fix is in progress with ETA 7th July. Blocked: customer (cannot generate labels for affected orders). Severity: high — customer has recurring orders hitting this limit, workaround is not viable due to third-party WMS dependency. Area: rate-shopping. (Comment #10, #11, #13, #16)

## Resolved Issues

1. **FedEx Freight Rate Failure Due to Sub-0.5 lb Line Item Weight** — Products with weight < 0.5 lbs were passing invalid weight values in the FedEx REST Freight rate request, triggering `LINEITEM.WEIGHT.REQUIRED` error. Fixed by automatically substituting 0.5 lbs as minimum weight when product weight is below threshold. (Comment #10, #11)

## Customer Context

- **Store**: 2af7c3-29.myshopify.com (US-based store)
- **Business type**: Multi-product retailer with mixed lightweight and heavyweight inventory; DaVinci WMS integration for warehouse management
- **Shipping volume**: 48 orders; $29 Popular Plan; active 427 days
- **Technical setup**: FedEx REST carrier; FedEx Freight LTL (Economy + Priority) configured via automation rules for orders over 150 lbs; partial fulfillment feature not enabled
- **Other context**: Customer explicitly cannot use the partial fulfillment workaround because enabling OnHold line items in Shopify would disrupt the DaVinci WMS order flow; waiting for the 15-line-item splitting enhancement (ETA: on or before 7th July)

## User Story

As a **multi-product retailer shipping heavyweight mixed-SKU orders through FedEx Freight with a DaVinci WMS integration**,
I want **FedEx Freight rate requests to handle orders of any size without hitting API line-item limits**,
so that **my warehouse team is not stuck manually splitting and rebooking shipments, delaying fulfillment for orders that used to ship cleanly before the FedEx REST migration**.

---

## Acceptance Criteria (Simple)

- [ ] FedEx Freight rate requests for orders with more than 15 unique line items succeed by automatically splitting into multiple sub-requests and aggregating the results
- [ ] Orders with line items weighing less than 0.5 lbs continue to use the 0.5 lb minimum weight floor (regression guard for the existing fix)
- [ ] Amy's store (2af7c3-29.myshopify.com) can generate FedEx Freight labels for order #3680 and similar high-line-item orders without manual intervention
- [ ] The line-item splitting logic correctly preserves special services (Liftgate Delivery, Call Before Delivery) across sub-requests
- [ ] FedEx Freight rate requests with 15 or fewer line items continue to work as a single request with no behavioral change

---

## Acceptance Criteria (Given/When/Then)

### Scenario: FedEx Freight rate request with more than 15 unique line items
- **Given** an order has 20 unique line items totaling over 150 lbs, with FedEx Freight LTL configured via automation rules
- **When** the app fetches FedEx Freight rates for this order
- **Then** the app splits the line items into sub-requests of 15 or fewer items each, fetches rates for each sub-request, and returns a combined rate to the merchant

### Scenario: Special services preserved across split requests
- **Given** a FedEx Freight order with 18 line items and Liftgate Delivery + Call Before Delivery enabled
- **When** the rate request is split into two sub-requests
- **Then** both sub-requests include the configured special services, and the combined rate reflects the correct freight charges

### Scenario: Orders at or below the 15 line-item threshold are unaffected
- **Given** a FedEx Freight order with exactly 15 or fewer unique line items
- **When** the app fetches rates
- **Then** the request is sent as a single API call with no splitting, and the behavior is identical to the current production flow
