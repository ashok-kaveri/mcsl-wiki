# Ticket #378589 — Orders 8885 and 8888 not syncing from WooCommerce (Second Shot Market)

- **Customer**: Mike Miller (secondshotmarket.com) — US WooCommerce store on Bluehost; UPS + Stamps.com USPS connected; labels were active through Jan 2026; $29 Popular plan; payment active
- **Product**: WooCommerce Multi Carrier Shipping
- **Duration**: 2026-03-13 → 2026-04-13 (ongoing, month-long L3 debug)

## Timeline & Key Phases

### Phase 1: Initial report and access setup (2026-03-13 → 2026-03-24)
- Customer reported that WooCommerce orders 8885 and 8888 are not syncing to the plugin; resync from WooCommerce did not help. Order 8885 had a PluginHive transaction ID but did not import (Comment #0).
- Internal profile: $29 Popular plan, UPS + Stamps.com USPS, label activity until Jan 2026, payment active; store sync is failing (Comments #1-2).
- Multiple rounds of WP-admin credential exchange; first login attempt failed, customer fixed the password, team regained access (Comments #3-10).

### Phase 2: First diagnostic — WP-JSON 403 (2026-03-24 → 2026-03-27)
- Team's diagnostic tool returned `Failed ( 403 :: Forbidden )` on the store's WP-JSON endpoint; webhook sync failing as a result; orders not importing (Comment #10).
- Customer checked with Bluehost — Bluehost sees webhooks delivering 200 OK, WP-JSON loads fine in browser, and reports no issues on their side; asked for PluginHive's server IPs and exact endpoints (Comments #11-12).
- Support re-shared WP-admin credentials and pushed for L3 help (Comments #13-14).
- L3 ticket filed (Comment #15): re-sync failing, webhook duplicates cleared but not coming back, WooCommerce-side resync succeeds but no webhooks visible; JSON fails in debugger but passes in browser. Slack thread opened.

### Phase 3: Cloudflare / bot-protection suspicion (2026-03-27 → 2026-04-08)
- Veena (L3): store responses include a Cloudflare "Just a moment…" challenge page, suggesting a bot/country block (Comments #16-17).
- Customer reported no recent security changes; only a newly installed MCP plugin after the failure started; granted permission to disable other plugins for troubleshooting (Comments #18-21).
- L3 asked customer to whitelist PluginHive's 5 server IPs: `159.89.150.142`, `165.227.28.252`, `138.68.13.182`, `165.227.48.244`, `157.245.169.250` (Comments #22-24).
- Customer misconfigured the whitelist, briefly took the site down, then restored access (Comments #25-30).
- Next round of debugging: store JSON then worked fine and plugin was up to date, but **PluginHive webhooks were missing / removed from WooCommerce**; L3 asked to re-add webhooks and retry import (Comments #31-32).

### Phase 4: Back to L3 — Cloudflare blocking US traffic (2026-04-06 → 2026-04-08)
- Veena tagged this as a store-level issue and debugged further (Comment #33).
- L3 finding (Athul, #34): Cloudflare security rules on the customer's store are blocking requests from PluginHive's US-based servers. Remediation path: Security → WAF → Custom Rules (disable any US/country block), or Security → Bots (turn off Bot Fight Mode / Super Bot Fight Mode). Public-facing reply sent (Comment #35). L3 also noted: if customer reports still broken, debug once more before re-escalating (Comment #36).

### Phase 5: Bluehost disables mod_security but 403 persists (2026-04-10 → 2026-04-13)
- Customer: Bluehost has disabled mod_security but the PluginHive debugger still returns 403. Attached debugger logs + Bluehost conversation transcript (Comment #37).
- Team acknowledged (Comment #38); re-opened L3 thread in Slack: `https://pluginhive.slack.com/archives/C02E15X8X0C/p1775792328767879?thread_ts=1774595992.171149&cid=C02E15X8X0C` (Comment #39).
- Veena: needs time to investigate (Comment #40).
- Latest: Mirdas asked team to follow up with dev and keep him posted (Comment #41).
