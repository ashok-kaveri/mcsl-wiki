## Open Issues

1. **Large batch creation: missing redirect to Label Batch page; misleading "Error in Batch Creation" message** — For 1000+ order batches, batch creation takes a long time and the app fails to auto-redirect to the Label Batch page (works correctly for small batches). During the delay, the app shows "Error in Batch Creation," which is misleading since the batch is often still processing successfully — confusing customers. Behavior captured in a video shared earlier. Blocked: engineering (dev_needed; q4_dec_end). Severity: P2 — directly affects high-volume customers' confidence in batch operation. Area: `label-generation`. (Comment #1)

2. **Failed labels appear as "Processing" in original batch AND in retry batch; retried carrier overwrites original batch** — Workflow: 2 of 100 labels fail; customer retries those 2 with a different carrier. Bug: the failed orders show "Processing" in BOTH the original batch and the new retry batch simultaneously; additionally, the retried carrier's name gets written back into the original batch, corrupting batch metadata. Dev team has previously noted possible printing inconsistencies tied to the same state-tracking bug. Blocked: engineering (dev_needed). Severity: P2 — data-correctness issue across retry workflow; affects audit/print paths. Area: `label-generation`. (Comment #1)

3. **Print mismatch: "Not in Ship" order had a generated label and was fulfilled in Shopify** — Order #45279880 was excluded from Label Print because its status was "Not in Ship" (per Deepak), yet a label was actually generated and Shopify shows the order as fulfilled. The Amazon tracking ID (https://track.amazon.in/tracking/369348516162) is valid. First observation of this state inconsistency; root cause unknown. Blocked: engineering (root-cause investigation). Severity: P2 — data integrity / fulfillment status drift; risks repeating across other orders silently. Area: `order-management`. (Comment #1)

4. **Cascading priority-based carrier routing across Amazon / Delhivery / XpressBees** — Customer's working pattern is: try Amazon first → switch failed orders to Delhivery → retry failed Amazon with Amazon again → fall back to XpressBees on remaining failures. They want this multi-step priority cascade automated rather than manual. Blocked: product/engineering (dev_needed; q4_dec_end). Severity: P2 feature-request — affects daily operations of an active high-volume merchant. Area: `rate-shopping`. (Comment #1)
