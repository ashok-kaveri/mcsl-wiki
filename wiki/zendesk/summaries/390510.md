# Ticket #390510 — Love of India: Order Import, Batch, Label, and Fulfillment Bugs

- **Customer**: Love of India (loveofindia.myshopify.com)
- **Product**: Shopify Multi Carrier Shipping Label
- **Duration**: 2026-05-18 to 2026-07-06 (ongoing)

## Timeline & Key Phases

### Phase 1: Initial Bug Report — Batch and Label Issues (2026-05-18)
- Veena documented 4 production issues: batch creation delays with misleading errors for 1000+ orders, failed labels persisting as "Processing" in original batch after retry, printing mismatch where "Not to Ship" order had a valid label/fulfillment, and carrier priority fallback logic request.

### Phase 2: Order Import Delays and Duplicate Labels (2026-05-21 to 2026-05-27)
- Customer reported order import inconsistencies: orders placed on a specific date don't appear unless date range is widened by +/- 1 day. Example: order #45283422 placed May 20 not visible in May 20 filter.
- Duplicate packages/labels reported: orders with 1 package randomly getting 2 packages and 2 labels during generation, in both large (100+) and small (3) batches.
- Archived orders with status=Archived and payment=Voided being imported as Processing.
- L3 noted: fulfilled orders appearing in "Not to Ship" view needed clearing; tag-assignment logic for label-failed orders discussed; archived order import needs investigation.
- Leading/trailing spaces in filter fixed (2026-05-28).

### Phase 3: Fixes Deployed Behind Toggles (2026-06-13 to 2026-06-17)
- Three fixes deployed (behind feature toggles, not yet active):
  1. Print mismatch: all labels in batch now included in print document.
  2. Fulfilled orders in "Not to Ship" view: handled, orders display in correct view.
  3. Large batch creation intermittent error: fixed.
- L2 reported tag-assigning logic doesn't fully match requirements — batch can contain both label success and failure orders, but tags should only apply when label creation fails across all 3 carriers.

### Phase 4: Fulfillment and Cancellation Issues (2026-06-12 to 2026-06-30)
- Order #45296293: label created June 10, fulfillment failed with "Order is already fulfilled" but Shopify timeline shows no fulfillment ever occurred. Customer had to refund because product was shipped but order never fulfilled in Shopify. Root cause: order was refunded before fulfillment attempt, app interpreted "nothing to fulfill" as "already fulfilled".
- Shopify cancellation overwriting Label Created status to Cancelled identified. Fix assigned to MCSL 384 release. Orders with existing labels should retain post-label status.
- Shipping zone postcode matching bug: app uses billing address postcode instead of shipping address postcode when checking shipping zone conditions. Order #45307249: shipping postcode 452005 (in zone), billing postcode 411015 (not in zone) — Amazon carrier skipped. Team unable to replicate; needs a processing order to verify.

### Phase 5: Consolidated Open Issues Review (2026-07-06)
- L3 compiled comprehensive pending/open cases list with all active issues.

## Open Issues

1. **Order import delay/inconsistency with date filter** — When filtering orders for a specific date, not all orders placed on that date appear. Missing orders only surface when the date range is adjusted by +/- 1 day. Blocked: engineering. Severity: High — causes orders to be missed during daily processing. Area: order-management. (Comments #58156453480985, #59744462577561)

2. **Fulfilled orders appearing in "Not to Ship" view** — Orders that have been fulfilled are showing in the Not to Ship section or having their status changed to Not to Ship. Fix was deployed behind toggle but L2 reported it may not be fully resolved. Blocked: engineering (toggle activation + verification). Severity: High — causes confusion in order processing. Area: order-management. (Comments #59744462577561, #59091975047833)

3. **Duplicate packages/labels generated** — Orders with 1 package randomly get 2 packages and 2 labels during label generation. Occurs in both large batches (100+) and small batches (as few as 3). Blocked: engineering. Severity: High — causes duplicate shipping charges and customer confusion. Area: label-generation. (Comments #58264813544345, #59744462577561)

4. **Archived and voided orders imported as Processing** — Orders with status=Archived and payment=Voided are being imported into the app under Processing status. These should not be imported. Blocked: engineering (investigation needed — some archived orders may be legitimately paid and need labels). Severity: Medium. Area: order-management. (Comments #58346274695577, #59744462577561)

5. **Fulfillment fails with "Order is already fulfilled" on never-fulfilled orders** — Order #45296293: label created, fulfillment attempted June 10, failed with "Order is already fulfilled" but Shopify timeline shows no fulfillment. Customer had to refund because product was physically shipped but never fulfilled in Shopify. Root cause: order was refunded before fulfillment attempt; app misinterpreted as already fulfilled. Blocked: engineering. Severity: Critical — caused direct financial loss to merchant. Area: label-generation. (Comments #58917699156377, #59744462577561)

6. **Shopify cancellation overwrites Label Created status to Cancelled** — When a Shopify order is cancelled, the app overwrites the order status to Cancelled even when a label was already created. Expected: orders in Label Created or later state should retain their post-label status. Fix identified, planned for MCSL 384. Blocked: engineering. Severity: High. Area: order-management. (Comments #59535169728281, #59566169729433)

7. **Shipping zone uses billing postcode instead of shipping postcode** — App checks billing address postcode against shipping zone postcode conditions instead of shipping address postcode. Order #45307249: shipping postcode 452005 (in zone for Amazon), billing postcode 411015 (not in zone) — Amazon carrier incorrectly skipped. Team unable to replicate; needs a processing order to verify. Blocked: engineering (reproduction). Severity: High — causes wrong carrier assignment. Area: carrier-config. (Comments #59568009421209, #59566169729433)

8. **Batch tag-assignment logic incomplete** — Tags should only be assigned to Shopify orders when label creation fails across all 3 carriers. Current implementation assigns tags to batch containing both success and failure orders, not matching the customer's requirement. Blocked: engineering. Severity: Medium. Area: label-generation. (Comment #59130193170585)

## Resolved Issues

1. **Leading/trailing spaces in filter not giving expected results** — Handled. (Comment #58387803260441)

2. **Print mismatch: labels missing from batch print document** — Fixed. All labels in batch now included in print document. Deployed behind toggle. (Comment #59091975047833)

3. **Large batch "Error in Batch Creation" intermittent error** — Fixed. Batches should create without error. Deployed behind toggle. (Comment #59091975047833)

4. **Fulfillment timing on order #45296293 explained** — Label generated at 2026-06-11T04:42:44Z (10:12 IST), fulfillment attempted at 2026-06-11T15:52:17Z (21:22 IST). Order was already refunded by fulfillment time; app interpreted nothing-to-fulfill as already-fulfilled. (Comment #58918816628377)
