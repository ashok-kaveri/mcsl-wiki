#7 DHL Express REST API Integration

P0-Critical Carrier

User Story

As a Shopify merchant using DHL Express,
I want MCSL to integrate the MyDHL REST API as a new carrier option,
So that I can use DHL Express services for label generation, rate queries, and tracking via the latest DHL API.

Scope & Impact

  • Affected merchants: All DHL Express users (~15-20% of active merchants)
  • Ticket signal: 1,093 DHL-related tickets in KB
  • Risk if delayed: Complete DHL Express outage for all merchants
  • Type: New Carrier Integration — DHL Express via REST API (treated as new carrier)

API Documentation & Technical References

Primary API: MyDHL API — DHL Express

Authentication: OAuth 2.0 (client_credentials grant) → replaces legacy XML basic auth

Base URL: https://express.api.dhl.com/mydhlapi (prod) | https://express.api.dhl.com/mydhlapi/test (sandbox)

Endpoints to implement:

  • POST /shipments — Label generation (replaces XML ShipmentRequest)
  • POST /rates — Rate query (replaces XML RateRequest)
  • GET /tracking?shipmentTrackingNumber={AWB} — Tracking
  • POST /pickups — Pickup scheduling
  • DELETE /shipments/{shipmentTrackingNumber} — Void label

Key differences from XML API:

  • JSON request/response (not XML SOAP)
  • OAuth 2.0 token with 3600s expiry (must implement token refresh)
  • Improved error codes with structured JSON error objects
  • Rate limiting: 100 requests/second (check X-RateLimit headers)
  • Label output formats: PDF, ZPL, PNG (base64 encoded in response)

Acceptance Criteria

  • AC-1: Customer can use DHL Express services to generate labels on REST API
  • AC-2: DHL Express rates are returned accurately at checkout via REST API
  • AC-3: OAuth 2.0 token auto-refreshes before expiry
  • AC-4: Tracking data displays all status events
  • AC-5: Pickup scheduling creates valid pickup requests
  • AC-6: Void label successfully cancels shipment
  • AC-7: Error messages display meaningful descriptions
  • AC-8: All DHL Express label sizes render correctly

Risks & Mitigations

  • Risk: OAuth token expiry causes intermittent failures
    Mitigation: Implement proactive token refresh (refresh at 80% of TTL)
  • Risk: Rate limiting during peak hours
    Mitigation: Implement retry with exponential backoff, queue rate requests

P1-High Carrier

I want FedEx One Rate pricing (including Ground) to automatically appear at checkout alongside standard FedEx rates,
So that my customers can choose flat-rate shipping options without me needing to enable any special setting.

  • Disable toggle: A new "Disable FedEx One Rate" toggle is needed in FedEx carrier settings for merchants who do NOT want One Rate shown.
  • When disabled, only standard FedEx rates appear (current behavior).
  • Ticket signal: 1,788 FedEx-related tickets in KB.
  • Affects: All FedEx-enabled merchants.

API Documentation & Technical References

Primary API: FedEx Rate API v1

Key parameter: rateRequestType: "ONE_RATE" in the rate request body

Eligible services: FedEx Ground, Express Saver, 2Day, Standard Overnight, Priority Overnight, First Overnight, Home Delivery

Packaging types to map:

  • FEDEX_ENVELOPE — up to 10 lbs
  • FEDEX_PAK — up to 50 lbs

  • FEDEX_MEDIUM_BOX — 13.25" x 11.5" x 2.375"
  • FEDEX_LARGE_BOX — 17.5" x 12.375" x 3"
  • FEDEX_EXTRA_LARGE_BOX — 11.875" x 11" x 10.75"

  • AC-2: A "Disable FedEx One Rate" toggle exists in FedEx carrier settings within the app.

    Mitigation: Include a one-time notification/changelog entry explaining the new default behavior and how to disable it.
  • Risk: One Rate packaging requirements (must use FedEx packaging) may confuse merchants.
    Mitigation: Clear tooltip on the disable toggle explaining when One Rate applies.

Critical Context — WooCommerce Reference Business Logic

The WooCommerce UPS plugin implements the correct mixed display behavior:

  • Step 1: When Simple Rate is enabled AND shipment is eligible, make TWO rate requests — one standard (general) and one with rateType: SIMPLE.
  • Step 2: Collect both response sets: $rates[$key]['general'][] and $rates[$key]['simple'][].
  • Step 3 (Merge): If same service exists in both, the general rate is REMOVED — Simple Rate pricing takes precedence.
  • Step 4: Final checkout displays merged list — Simple Rate priced services + general-only services.

In short: Simple Rate replaces the price of eligible services but does NOT hide ineligible services.

#42 Bundled Products Support

P1-High Ecommerce

User Story

As a Shopify merchant selling product bundles,
I want the app to correctly identify, weigh, price, and package bundled products,
So that shipping rates are accurate, labels have correct declared values, and packing slips show the right item count.

Scope & Impact

  • Two cards: Bundled Products Foundation + Incorrect Product Count & Value Bug
  • Zendesk: #260001, #372492
  • Two bundle types: (1) Prepacked — single box, one weight/dimension. (2) Individual — separate products, each with own weight/dimensions.
  • Bug: Same product in bundle AND individual → value cross-contamination, wrong packing slip counts.
  • Affects: Merchants using Shopify native bundles. Third-party bundle apps (Bold Bundles, Bundler) to be added in a follow-up phase.

Context — Two Cards

Card #1 — Bundled Products Foundation:

  • Prepacked: Ships together in single box. Weight and dimensions configured for all products together.
  • Individual: Products ship separately. Weight and dimensions configured separately per product.
  • Changes needed: Product import, line item processing for rates, Order import, Order processing, Packaging.

Card #2 — Incorrect Product Count and Value (#3853):

  • Bundled product value incorrectly applied to both bundled and individual items.
  • Packing Slip total does not include individual product correctly.
  • Manual package edit overrides individual product values with bundle values.

Reference Implementation — WooCommerce UPS Bundle Product Support

The WooCommerce UPS plugin has a proven bundled product implementation (func-ph-ups-woocommerce-bundle-product-support.php) that provides a reference pattern:

  • Bundle detection: Uses _bundled_item_needs_shipping meta field to determine if bundled item requires shipping (skip virtual/digital items).
  • Pricing mode: Checks _bundled_item_priced_individually — if yes, calculates per-item price from line total / quantity. If not, uses bundle parent price.
  • Quantity aggregation: Groups identical products by product_id — if same product appears in multiple bundles, quantities are summed (prevents duplicate entries).
  • Commercial invoice: ph_update_invoice_products_for_product_bundle() iterates $product->get_bundled_items() to include child items on customs invoices with correct values, HS codes, and country of manufacture.
  • Setting: include_child_products_for_bundle — controls whether child items appear on invoices.

Key takeaway for MCSL: The Shopify implementation must handle the same concerns — shipping vs non-shipping items, individual vs flat pricing, quantity aggregation, and commercial invoice line items — but using Shopify's bundle metadata format (line_item.properties) instead of WooCommerce meta fields.

API Documentation & Technical References

Primary API: Shopify Product API

Bundle detection: product.productType == "bundle" or components field present

Order line items: Bundle metadata in properties or line_item.properties

Phase 1 scope: Shopify native bundles only. Third-party bundle apps (Bold Bundles, Bundler) in follow-up phase.

WooCommerce reference: ph-woocommerce-ups/.../func-ph-ups-woocommerce-bundle-product-support.php

Acceptance Criteria

  • AC-1: Prepacked bundles treated as single unit for shipping calculation.
  • AC-2: Individual-product bundles treated as separate items.
  • AC-3: Product import correctly identifies bundle type and saves appropriate data.
  • AC-4: Rate calculation uses correct weight per bundle type.
  • AC-5: Same product in bundle AND as individual: no value cross-contamination.
  • AC-6: Packing Slip: correct entries, quantities, and total item count.
  • AC-7: Manual package edit does NOT override individual values with bundle values.
  • AC-8: Order import correctly parses Shopify native bundle metadata from Shopify API.
  • AC-9: International shipments: correct declared values per item on customs documents.

Risks & Mitigations

  • Risk: Shopify native bundle metadata format may evolve as Shopify iterates on the feature.
    Mitigation: Build against current Shopify bundle API docs. Design for extensibility so third-party bundle apps can be added in a follow-up phase.
  • Risk: Prepacked vs individual detection may be ambiguous.
    Mitigation: Add clear UI indicator showing how each bundle was interpreted. Reference WooCommerce approach: _bundled_item_needs_shipping + _bundled_item_priced_individually meta fields.
  • Risk: Real-world edge cases from support tickets — merchants with mixed bundles (e.g., food + merchandise needing different packaging), bundles with max-items-per-box limits, single products splitting across multiple boxes.
    Mitigation: Ensure packaging logic respects per-item-type constraints. Test with multi-category bundles.
  • Risk: Commercial invoices for international shipments may list bundle parent instead of child items.
    Mitigation: Follow WooCommerce UPS pattern — iterate child items for customs documents with correct values, HS codes, and country of manufacture.
